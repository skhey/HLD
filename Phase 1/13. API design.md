# API Design: RESTful Principles, Pagination, Idempotency Keys, and Versioning

Designing robust, scalable, and developer‑friendly APIs is a cornerstone of modern software architecture. This guide covers four essential aspects of API design that frequently appear in system design interviews and real‑world implementations: **RESTful principles**, **pagination**, **idempotency keys**, and **versioning**. Mastering these concepts will help you create APIs that are intuitive, reliable, and easy to evolve.

---

## 1️⃣ RESTful Principles

REST (Representational State Transfer) is an architectural style for designing networked applications. It was introduced by Roy Fielding in his doctoral dissertation and has become the de facto standard for web APIs.

### Core Constraints of REST

| Constraint | Description |
|------------|-------------|
| **Client‑Server** | Separation of concerns: the client (UI) and server (data storage) evolve independently. |
| **Stateless** | Each request from client to server must contain all information needed to understand the request. The server does not store any client context between requests. |
| **Cacheable** | Responses must explicitly indicate whether they are cacheable or not, improving performance and reducing server load. |
| **Uniform Interface** | A consistent way to interact with resources. This is the most defining constraint and includes: |
| • **Resource identification in requests** (e.g., URIs) |
| • **Resource manipulation through representations** (e.g., JSON) |
| • **Self‑descriptive messages** (each message contains enough info to process it) |
| • **HATEOAS** (Hypermedia as the Engine of Application State) – clients navigate via hyperlinks in responses. |
| **Layered System** | The architecture can be composed of hierarchical layers (e.g., load balancers, caches), and each component only knows about the immediate layer it interacts with. |
| **Code on Demand (optional)** | Servers can temporarily extend client functionality by transferring executable code (e.g., JavaScript). |

### Resource‑Oriented Design
In REST, everything is a **resource** (e.g., users, orders, products). Resources are identified by URIs (e.g., `/users/123`). Clients interact with resources using standard HTTP methods:

| Method | Purpose | Example | Idempotent? |
|--------|---------|---------|-------------|
| **GET** | Retrieve a representation of a resource. | `GET /users/123` | Yes |
| **POST** | Create a new resource (or perform an operation). | `POST /users` with body | No |
| **PUT** | Replace an entire resource. | `PUT /users/123` with body | Yes |
| **PATCH** | Partially update a resource. | `PATCH /users/123` with changes | No (can be made idempotent with care) |
| **DELETE** | Remove a resource. | `DELETE /users/123` | Yes |

### HTTP Status Codes
Use appropriate status codes to indicate the result of an operation:

| Code Range | Meaning | Examples |
|------------|---------|----------|
| **2xx** | Success | 200 OK, 201 Created, 204 No Content |
| **3xx** | Redirection | 301 Moved Permanently, 304 Not Modified |
| **4xx** | Client Error | 400 Bad Request, 401 Unauthorized, 404 Not Found, 429 Too Many Requests |
| **5xx** | Server Error | 500 Internal Server Error, 503 Service Unavailable |

### Example RESTful API Design
```
GET    /users              → List users (with pagination)
POST   /users              → Create a new user
GET    /users/{id}         → Get user details
PUT    /users/{id}         → Replace entire user
PATCH  /users/{id}         → Update user (partial)
DELETE /users/{id}         → Delete user
GET    /users/{id}/orders  → List orders of a user
```

## 2️⃣ Pagination

When an API endpoint can return a large number of items, pagination is essential to avoid overwhelming the client or the server. It splits the result set into manageable pages.

### Why Paginate?
- **Performance:** Reduces response size and database load.
- **User experience:** Clients can fetch data incrementally (e.g., infinite scroll, "next page" buttons).
- **Resource protection:** Prevents accidental denial‑of‑service by limiting the amount of data returned.

### Common Pagination Strategies

| Strategy | How It Works | Example | Pros | Cons |
|----------|--------------|---------|------|------|
| **Offset‑Based (Limit/Offset)** | Client specifies `offset` (number of items to skip) and `limit` (page size). | `GET /items?offset=20&limit=10` | Simple, easy to implement. | Inefficient for large offsets (database still scans skipped rows); inconsistent if items are inserted/deleted between page fetches. |
| **Cursor‑Based** | Server returns a cursor (opaque token) that points to the next page. Client includes that cursor in subsequent requests. | `GET /items?cursor=abc123&limit=10` | Efficient, stable under data changes (cursor often points to a specific item). | More complex to implement; cursor encoding/decoding required. |
| **Keyset Pagination (Seek Method)** | Client provides the last seen value of a sorted column (e.g., `id`, `created_at`). | `GET /items?after_id=100&limit=10` | Very efficient (uses indexed column). | Tied to a specific sort order; requires unique, sortable column. |
| **Page‑Based** | Client requests a specific page number. | `GET /items?page=3&size=10` | Simple and familiar. | Same offset inefficiency; pages shift if items are added/deleted. |

### Best Practices
- Always set a **maximum page size** to prevent abuse (e.g., `limit <= 100`).
- Provide default values (e.g., `page=1`, `size=20`).
- Include pagination metadata in the response, such as:
  ```json
  {
    "data": [...],
    "pagination": {
      "total": 1234,
      "page": 3,
      "per_page": 10,
      "next_page": 4,
      "prev_page": 2,
      "next_url": "/items?page=4&size=10",
      "prev_url": "/items?page=2&size=10"
    }
  }
  ```
- For cursor‑based pagination, return the next cursor in the response:
  ```json
  {
    "data": [...],
    "next_cursor": "abc123xyz"
  }
  ```
- Use **Link headers** (RFC 5988) for standardised pagination links:
  ```
  Link: <https://api.example.com/items?cursor=abc123>; rel="next",
        <https://api.example.com/items?cursor=def456>; rel="prev"
  ```

### Example: Offset‑Based Pagination in REST
```
GET /api/products?offset=0&limit=20
```
Response:
```json
{
  "items": [...],
  "total": 105,
  "offset": 0,
  "limit": 20,
  "next": "/api/products?offset=20&limit=20",
  "prev": null
}
```

### Cursor‑Based Example (using `id` as cursor)
```
GET /api/products?cursor=eyJpZCI6MTAwfQ==&limit=20
```
The cursor is often a base64‑encoded JSON containing the last seen value. The server decodes it and constructs a query like `WHERE id > 100 ORDER BY id LIMIT 20`.

## 3️⃣ Idempotency Keys

Idempotency ensures that performing the same operation multiple times has the same effect as doing it once. In APIs, this is crucial for operations that can be retried (e.g., due to network timeouts) without causing unintended side effects (like double charging).

### Why Idempotency Matters
- Clients may retry requests when they don't receive a response (timeouts, network issues).
- Without idempotency, a retry could create duplicate resources or perform the same action twice.

### Idempotent Methods in HTTP
- **GET, PUT, DELETE, HEAD, OPTIONS** are defined as idempotent by HTTP specification.
- **POST** and **PATCH** are not inherently idempotent. However, you can make them idempotent using an **idempotency key**.

### How Idempotency Keys Work
1. **Client generates a unique key** (e.g., UUID v4) for each operation it intends to perform.
2. **Client includes the key** in the request header (e.g., `Idempotency-Key: 123e4567-e89b-12d3-a456-426614174000`).
3. **Server stores the key** along with the response (or the fact that the operation was processed) for a certain retention period.
4. If the same key is received again, the server returns the stored response without re‑executing the operation.

### Implementation Details
- **Storage:** Use a fast, persistent store like Redis or a database table with a TTL.
- **Scope:** Idempotency keys are typically scoped to a user or API key to prevent collisions.
- **Response caching:** Store the complete response (status code, headers, body) so that the same response can be returned on duplicate requests.
- **Expiry:** Keys should expire after a reasonable time (e.g., 24 hours) to avoid unbounded storage growth.

### Example: Stripe's Idempotency Key
Stripe requires an idempotency key for any request that could create or update resources (e.g., creating a charge). If you retry the same request with the same key, Stripe returns the same result.

```
POST /v1/charges
Idempotency-Key: 123e4567-e89b-12d3-a456-426614174000
{
  "amount": 2000,
  "currency": "usd",
  "source": "tok_visa"
}
```

### Making a POST Endpoint Idempotent (Pseudocode)
```python
def create_order(request):
    key = request.headers.get('Idempotency-Key')
    if not key:
        return error("Idempotency-Key required", 400)

    # Check if we've seen this key before
    cached = cache.get(key)
    if cached:
        return cached['response'], cached['status']

    # Process the request (e.g., create order)
    order = create_order_in_db(request.body)

    response = {"order_id": order.id, "status": "created"}
    # Cache the response for 24 hours
    cache.set(key, {'response': response, 'status': 201}, ex=86400)

    return response, 201
```

### Important Considerations
- The idempotency key must be **unique per client and operation**. Clients are responsible for generating them.
- The server should **not** reuse keys across different operations.
- If a request with an existing key differs from the original (different request body), the server should return an error (e.g., 422 Unprocessable Entity) because idempotency only guarantees same effect for identical requests.

## 4️⃣ API Versioning

As your API evolves, you need to introduce changes without breaking existing clients. Versioning allows you to release new features or modify existing ones while maintaining backward compatibility.

### Why Version?
- Add new fields or endpoints.
- Change existing field semantics or remove deprecated fields.
- Fix bugs that could alter expected behavior.
- Allow time for clients to migrate.

### Versioning Strategies

| Strategy | How It Works | Example | Pros | Cons |
|----------|--------------|---------|------|------|
| **URI Path** | Include version in the URL path. | `GET /v1/users`, `GET /v2/users` | Very explicit; easy to route at the load balancer. | URI changes require clients to update; can lead to code duplication on server. |
| **Query Parameter** | Version specified as a query param. | `GET /users?version=1` | Simple, URL stays the same. | Easy to forget; caching may be affected if version not part of cache key. |
| **Custom Header** | Use a custom header (e.g., `Accept-Version`). | `GET /users` with header `Accept-Version: 1` | Keeps URLs clean. | Requires clients to set headers; not visible in logs/URIs. |
| **Content Negotiation (Accept header)** | Use `Accept` header with custom media type. | `Accept: application/vnd.myapi.v1+json` | Leverages HTTP semantics; version tied to content type. | More complex to parse; requires careful design of media types. |

### Best Practices
- **Start with versioning from day one.** Even if you don't need it immediately, it's easier to add later if you have a plan.
- **Choose a strategy that fits your ecosystem.** For public APIs, URI path is most common and explicit.
- **Keep backward compatibility within a version.** Don't break existing clients when you add features; only introduce breaking changes in a new version.
- **Document your versioning policy** clearly, including deprecation timelines.
- **Support multiple versions simultaneously** for a transition period, then gracefully retire old versions.

### Example: URI Path Versioning
```
GET /api/v1/users/123
GET /api/v2/users/123
```

### Example: Content Negotiation with Custom Media Type
```
GET /users/123
Accept: application/vnd.mycompany.v1+json
```
The server responds with the appropriate representation.

### Handling Deprecation
- Communicate deprecation via response headers (e.g., `Deprecation: true`, `Sunset: Sat, 1 Jan 2025 23:59:59 GMT`).
- Return a `Warning` header or include a deprecation notice in the response body.
- Provide clear migration guides and timelines.

## ✅ Summary of Key Takeaways

| Topic | Key Points |
|-------|------------|
| **RESTful Principles** | Stateless, resource‑oriented, use HTTP methods appropriately, leverage status codes, consider HATEOAS for discoverability. |
| **Pagination** | Essential for large datasets. Choose offset, cursor, or keyset pagination based on consistency and performance needs. Always include metadata and limit max page size. |
| **Idempotency Keys** | Make non‑idempotent operations (POST) safe for retries. Use unique client‑generated keys, store responses, and expire after a reasonable period. Critical for payments and order creation. |
| **API Versioning** | Plan for evolution. Use URI path for simplicity, or content negotiation for more flexibility. Support backward compatibility and communicate deprecation clearly. |

Mastering these API design concepts will enable you to build interfaces that are robust, developer‑friendly, and ready to scale. They are frequently discussed in system design interviews and are essential for any software architect.
