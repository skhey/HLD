# SQL Databases: ACID, Indexing, Replication, and Sharding

This guide covers the core concepts of SQL (relational) databases that are essential for high‑level design. Understanding these will help you make informed decisions about data storage, consistency, scalability, and performance in large‑scale systems.

---

## 1️⃣ ACID Properties

ACID is a set of properties that guarantee reliable processing of database transactions. They are crucial for systems where data integrity is paramount (e.g., banking, e‑commerce).

| Property | Description | Example / Analogy |
|----------|-------------|-------------------|
| **Atomicity** | A transaction is treated as a single, indivisible unit. It either executes completely or not at all (all‑or‑nothing). | Transferring $100 from A to B: either both the debit and credit happen, or neither does. If the system crashes mid‑way, the transaction is rolled back. |
| **Consistency** | A transaction brings the database from one valid state to another, preserving all defined rules (constraints, triggers, cascades). No transaction can violate data integrity. | After the transfer, the total money in accounts A and B remains unchanged. Any foreign key, unique, or check constraints must be satisfied. |
| **Isolation** | Concurrent transactions are executed in isolation, as if they were serialized (one after the other). Intermediate states are invisible to other transactions. | Two users trying to buy the last concert ticket: the database ensures that only one succeeds, and the other sees a consistent state (ticket sold). |
| **Durability** | Once a transaction is committed, its changes persist even in the event of a system failure (e.g., power outage, crash). Data is written to non‑volatile storage. | After you receive a payment confirmation, the record is safely stored on disk and won’t disappear even if the server reboots. |

### Isolation Levels (ANSI SQL)
To balance strictness and performance, SQL offers multiple isolation levels:

| Level | Phenomena Prevented | Use Case |
|-------|---------------------|----------|
| **Read Uncommitted** | None (dirty reads possible) | Rarely used; high performance but no consistency. |
| **Read Committed** | Dirty reads | Default in many databases; ensures you only read committed data. |
| **Repeatable Read** | Dirty reads, non‑repeatable reads | Prevents changes to rows you’ve already read during the transaction. |
| **Serializable** | All (dirty reads, non‑repeatable reads, phantom reads) | Strongest isolation, but slowest; used for critical operations (e.g., booking). |

**Phenomena:**
- *Dirty read*: Reading uncommitted data from another transaction.
- *Non‑repeatable read*: Getting different values when reading the same row twice within a transaction because another transaction updated it.
- *Phantom read*: New rows appear or disappear when re‑executing a query due to another transaction’s inserts/deletes.

## 2️⃣ Indexing

Indexes are data structures that speed up data retrieval at the cost of additional storage and slower writes. They are analogous to the index in a book – you can quickly find a topic without scanning every page.

### B‑Tree Index
The most common index type, used by default in nearly all SQL databases (e.g., InnoDB in MySQL, PostgreSQL).

**How It Works:**
- A balanced tree structure where each node contains keys and pointers.
- Leaf nodes contain the actual data (or pointers to data rows) and are linked for efficient range scans.
- Search, insert, delete, and range queries are O(log n).

**Example:**
```sql
CREATE INDEX idx_users_email ON users(email);
```
Now a query `SELECT * FROM users WHERE email = 'alice@example.com'` can use the B‑Tree index instead of a full table scan.

**Best for:**
- Equality and range queries (`=`, `<`, `>`, `BETWEEN`, `LIKE` with prefix).
- Sorting and grouping.
- Multi‑column indexes (composite) can cover queries if columns are chosen wisely.

### Hash Index
Uses a hash table to map keys to row locations.

**How It Works:**
- Applies a hash function to the indexed column; the result points directly to the data bucket.
- Lookups are O(1) on average.

**Example:**
```sql
-- In PostgreSQL, you can specify hash index
CREATE INDEX idx_users_email_hash ON users USING HASH (email);
```

**Limitations:**
- Only supports equality comparisons (`=`), not ranges (`<`, `>`, `BETWEEN`).
- Not ordered, so cannot be used for sorting or prefix searches.
- Vulnerable to hash collisions (though handled).
- Not durable in some implementations (e.g., MySQL `MEMORY` engine).

**Best for:**
- High‑throughput key‑value lookups where range queries are not needed.

### Comparison: B‑Tree vs. Hash

| Feature | B‑Tree | Hash |
|---------|--------|------|
| **Lookup speed** | O(log n) | O(1) (average) |
| **Range queries** | ✅ Yes | ❌ No |
| **Sorting support** | ✅ Yes (ordered) | ❌ No |
| **Composite keys** | ✅ Supports multi‑column | ❌ Typically single‑column |
| **Storage overhead** | Moderate (tree structure) | Low (hash table) |
| **Concurrency** | Good (page‑level locking) | Can have contention on buckets |

**In practice:** B‑Trees are the default and work for most use cases. Hash indexes are niche and used in specialized engines (e.g., MySQL `MEMORY` tables, PostgreSQL hash indexes).

### Indexing Strategies for System Design
- **Covering index:** An index that contains all columns needed for a query, avoiding a trip to the table.
- **Composite index order:** Put the most selective columns first.
- **Avoid over‑indexing:** Each index slows down writes (INSERT/UPDATE/DELETE) and consumes disk space.

## 3️⃣ Replication

Replication is the process of copying data from one database server to others. It improves availability, read scalability, and disaster recovery.

### Master‑Slave (Leader‑Follower) Replication
- One node is the **master** (leader) that handles all writes.
- One or more **slaves** (followers) replicate data from the master and serve read queries.
- Writes are propagated either **synchronously** (slave confirms before commit) or **asynchronously** (master commits, then propagates later).

**Architecture Diagram:**
```
        Clients
           │
           ▼
     [Load Balancer]
        ╱       ╲
       ▼         ▼
    [Master] → [Slave 1] → [Slave 2] (replication)
    (writes)   (reads)      (reads)
```

**Advantages:**
- Read scalability: distribute read load across many slaves.
- High availability: if master fails, a slave can be promoted.
- Analytics/backup offload: run heavy reports on a slave without affecting master.

**Challenges:**
- **Replication lag:** Asynchronous slaves may serve stale data.
- **Write bottleneck:** All writes still go to a single master.
- **Failover complexity:** Promoting a slave requires handling potential data loss if replication wasn't synchronous.

### Synchronous vs. Asynchronous Replication

| Mode | Write Acknowledgment | Pros | Cons |
|------|----------------------|------|------|
| **Synchronous** | Master waits for at least one slave to confirm write before committing. | No data loss if master fails; slave is always consistent. | Slower writes, reduced availability if slave is down. |
| **Asynchronous** | Master commits immediately; slaves replicate later. | Fast writes, master unaffected by slave issues. | Possible data loss if master crashes before replication. |

### Multi‑Master (Leader‑Leader) Replication
- Multiple nodes accept writes (no single master).
- Each node replicates its changes to the others.
- Conflict resolution is necessary when two nodes update the same data concurrently.

**Conflict Resolution Strategies:**
- **Last‑write‑wins (LWW):** Use timestamps or version numbers; simplest but may lose data.
- **Merge / CRDTs:** Data structures that automatically merge (e.g., counters, sets).
- **Custom application logic:** The application resolves conflicts by reading all versions.

**Use Cases:**
- Geographic distribution (writes in multiple regions).
- High‑availability where a single master would be a bottleneck or SPOF.
- Collaborative applications (e.g., Google Docs style).

**Challenges:**
- Conflict resolution complexity.
- Potential for write amplification and higher latency (cross‑region sync).
- Most SQL databases don't handle multi‑master well out‑of‑box; often use custom solutions (e.g., MySQL with Tungsten, or Galera Cluster for synchronous multi‑master).

## 4️⃣ Sharding (Horizontal Partitioning)

Sharding splits a large database into smaller, independent databases called **shards**. Each shard holds a subset of the data and runs on a separate server. This is the primary way to scale writes and storage beyond a single node.

### How Sharding Works
- A **shard key** is chosen (e.g., `user_id`, `country`, `order_id`).
- Data is distributed across shards based on the shard key (e.g., hash modulo, range).
- Each shard is a standalone database; together they form the logical whole.

**Example:** An e‑commerce site with 100 million users could be sharded by `user_id`:
- Shard 0: users 1–10M
- Shard 1: users 10M–20M, etc.

### Sharding Strategies

| Strategy | Description | Pros | Cons |
|----------|-------------|------|------|
| **Key‑Based (Hash)** | Apply a hash function to the shard key, then modulo by number of shards. | Even distribution, simple. | Adding/removing shards requires rehashing (expensive). |
| **Range‑Based** | Partition by contiguous ranges of the shard key (e.g., dates, alphabetical). | Easy to add new ranges; good for time‑series. | Can cause hot spots if distribution is uneven. |
| **Directory‑Based** | Maintain a lookup table that maps keys to shards. | Flexible; easy to rebalance. | Lookup table becomes a bottleneck or SPOF. |

### Choosing a Shard Key
- **High cardinality:** The key should have many distinct values to avoid uneven distribution.
- **Even distribution:** Writes and reads should be spread across shards.
- **Query patterns:** Ideally, most queries include the shard key so they can be routed to a single shard. Cross‑shard queries are expensive.

### Challenges of Sharding
- **Cross‑shard joins:** Queries that need data from multiple shards are complex and slow. You may need to denormalize or use application‑level aggregation.
- **Distributed transactions:** ACID across shards is hard; often you design around it.
- **Rebalancing:** When you add or remove shards, data must be migrated. This is a complex operation.
- **Backup/restore:** Must coordinate across all shards.

### Sharding vs. Partitioning (within a database)
- **Partitioning** (sometimes called "table partitioning") splits a table within the same database instance (e.g., by date ranges). It's transparent to the application.
- **Sharding** splits the data across separate physical servers, requiring application or middleware awareness.

## 5️⃣ Putting It All Together – Example: Scalable E‑Commerce Database

Consider an e‑commerce platform with millions of users and orders.

- **ACID** ensures order integrity: when a user places an order, inventory is decremented atomically, and payment is recorded.
- **Indexes:**
  - B‑Tree index on `user_id` for fast user lookups.
  - B‑Tree index on `order_date` for reporting.
  - Hash index on `session_id` (if using memory table) for quick session retrieval.
- **Replication:**
  - Master handles all writes (orders, inventory updates).
  - Multiple read‑only slaves serve product catalog queries, order history reads.
  - Asynchronous replication is acceptable for catalog; synchronous for critical inventory (if needed).
- **Sharding:**
  - Orders table is sharded by `user_id` to distribute write load and keep a user's orders on one shard.
  - Product catalog might be read‑only and replicated globally, not sharded, or sharded by product category.

---

## 6️⃣ Key Takeaways for HLD Interviews

- **ACID** is non‑negotiable for financial/transactional systems.
- **Indexes** speed up reads but slow writes; choose the right type (B‑Tree vs. Hash) based on query patterns.
- **Replication** improves read scalability and availability; understand the trade‑off between synchronous (consistency) and asynchronous (performance).
- **Sharding** is essential for write‑scalability beyond a single node; carefully pick a shard key that aligns with query patterns.
- **Always discuss trade‑offs:** consistency vs. latency, write throughput vs. read complexity, ease of implementation vs. future scaling.

Mastering these concepts will enable you to design robust, scalable data layers for any application.
