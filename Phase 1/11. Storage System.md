# Storage Systems: Object Storage, Block Storage, File Storage, and Blob Storage

In modern distributed systems, choosing the right storage type is critical for performance, scalability, and cost. Each storage model – **block**, **file**, and **object** – serves different use cases and has distinct characteristics. This guide explains each type, provides real‑world examples, and helps you decide which to use in your high‑level designs.

---

## 1️⃣ Overview of Storage Models

| Storage Type | Abstraction Level | Access Pattern | Typical Protocols | Examples |
|--------------|-------------------|----------------|-------------------|----------|
| **Block Storage** | Low‑level (raw disk) | Read/write in fixed‑size blocks | iSCSI, Fibre Channel | AWS EBS, Azure Disk Storage, GCP Persistent Disk |
| **File Storage** | File and directory hierarchy | File operations (open, read, write) | NFS, SMB/CIFS | AWS EFS, Azure Files, GCP Filestore |
| **Object Storage** | Flat namespace of objects with metadata | HTTP/HTTPS (REST API) | S3 API, Swift | AWS S3, Azure Blob Storage, GCP Cloud Storage |

Each model is optimized for different workloads. Understanding their differences is essential for designing scalable, cost‑effective storage architectures.

---

## 2️⃣ Block Storage

Block storage presents raw storage volumes to servers, much like a physical hard drive. Data is divided into fixed‑sized blocks, each with its own address. The operating system or application manages the file system on top of these blocks.

### Key Characteristics
- **Low‑level:** Applications access blocks directly (via a file system or raw I/O).
- **High performance:** Very low latency, high IOPS, suitable for databases.
- **Persistent:** Data persists independently of the server's lifecycle (in cloud volumes).
- **Typically attached to a single server:** Shared access is possible but complex (requires cluster file systems).

### How It Works
- A block storage volume (e.g., AWS EBS) is attached to a virtual machine (EC2 instance).
- The instance formats the volume with a file system (ext4, XFS) and mounts it.
- Reads and writes are performed at the block level, cached by the OS.

### Examples
- **AWS Elastic Block Store (EBS):** Provides persistent block storage for EC2. Offers different volume types (gp3, io2) for balanced or high‑performance needs.
- **Azure Disk Storage:** Managed disks for Azure VMs.
- **GCP Persistent Disk:** Durable block storage for Compute Engine.
- **SAN (Storage Area Network):** Traditional enterprise block storage.

### Use Cases
- **Databases (relational and NoSQL):** Require low latency and high IOPS (e.g., MySQL, PostgreSQL, Cassandra on EBS).
- **Virtual machine file systems:** OS boot volumes, application data.
- **High‑performance computing:** Workloads needing direct disk access.
- **Legacy applications:** Designed to run on local disks, can be lifted‑and‑shifted to cloud block storage.

### Trade‑Offs
- **Limited sharing:** Cannot be easily attached to multiple servers (except via cluster FS like OCFS2).
- **Cost:** Generally more expensive per GB than object storage.
- **Management:** Requires provisioning, snapshot management, and backup strategies.

## 3️⃣ File Storage

File storage provides a shared file system interface, allowing multiple clients to access and modify files over a network. It presents a hierarchical directory structure, with files and folders, and supports standard file operations (open, read, write, lock).

### Key Characteristics
- **Shared access:** Multiple servers (or users) can mount the same file system concurrently.
- **Hierarchical namespace:** Organised as directories and files.
- **POSIX‑like semantics:** Supports file locking, permissions, and sometimes extended attributes.
- **Network protocols:** Typically NFS (Network File System) or SMB (Server Message Block).

### How It Works
- A file storage server (e.g., AWS EFS) exports a file system over NFS.
- Clients (EC2 instances) mount the file system using the NFS client.
- All clients see the same files and can read/write concurrently (with locking for consistency).

### Examples
- **AWS Elastic File System (EFS):** Managed NFS file system that scales automatically.
- **Azure Files:** Managed SMB and NFS file shares.
- **GCP Filestore:** Managed NFS file storage.
- **Traditional NAS (Network Attached Storage):** On‑premises file servers.

### Use Cases
- **Shared home directories:** User profiles accessible from multiple application servers.
- **Content management systems:** Shared storage for uploaded files (images, documents).
- **Web server farms:** Shared static assets (CSS, JS) served by multiple web servers.
- **Big data / analytics:** Shared scratch space for Spark or Hadoop clusters.
- **Lift‑and‑shift:** Migrating applications that rely on shared file systems (e.g., legacy ERP).

### Trade‑Offs
- **Scalability limits:** Traditional file servers have capacity and performance ceilings; cloud file services like EFS scale automatically but still have limits on IOPS and throughput.
- **Cost:** Often more expensive than object storage for large volumes of infrequently accessed data.
- **Locking overhead:** Concurrent writes to the same file require coordination, which can become a bottleneck.

## 4️⃣ Object Storage (and Blob Storage)

Object storage manages data as **objects**, each consisting of:
- The data itself (unstructured, could be anything).
- Metadata (key‑value pairs describing the object).
- A globally unique identifier (e.g., a URL).

Objects are stored in a flat namespace (buckets or containers). There is no hierarchy; you can simulate folders using naming conventions (e.g., `photos/2025/vacation.jpg`), but the storage system itself treats it as a single flat list.

### Key Characteristics
- **HTTP‑based API:** RESTful access (GET, PUT, DELETE) over HTTP/HTTPS.
- **Massive scalability:** Designed to store exabytes of data across many servers.
- **Durability:** Data is automatically replicated across multiple devices and locations.
- **Metadata rich:** You can attach custom metadata to each object.
- **Eventually consistent** (in some implementations) for certain operations.

### How It Works
- A client (application) makes an HTTP PUT request to upload an object to a bucket.
- The object is stored, often erasure‑coded or replicated, and a URL is returned.
- Retrieval is via HTTP GET with the object's URL.
- Metadata can be included in headers or as separate attributes.

### Examples
- **AWS Simple Storage Service (S3):** The de facto standard object storage. Buckets, objects, versioning, lifecycle policies.
- **Azure Blob Storage:** Microsoft's object storage. Containers hold blobs (block blobs, append blobs, page blobs). Often called "blob storage" – essentially object storage.
- **GCP Cloud Storage:** Similar, with buckets and objects.
- **OpenStack Swift:** Open‑source object storage.
- **MinIO:** Self‑hosted S3‑compatible object storage.

### Blob Storage Clarification
"Blob" stands for Binary Large Object. **Azure Blob Storage** is a specific implementation of object storage. In common usage, "blob storage" is often used synonymously with object storage, especially in Microsoft ecosystems. So, for practical purposes, **blob storage = object storage**.

### Use Cases
- **Static website hosting:** Store HTML, CSS, JS, and serve directly via HTTP.
- **Backup and archive:** Durable, low‑cost storage for backups (with lifecycle policies to move to cold tiers).
- **Big data / data lakes:** Store raw data (logs, sensor data, Parquet files) for processing with Spark, Presto, etc.
- **Media storage:** Images, videos, audio files, often served via CDN.
- **Content distribution:** Combined with CDN for global delivery.
- **Disaster recovery:** Replicate objects across regions.

### Trade‑Offs
- **Not a file system:** Cannot mount as a drive; requires HTTP API.
- **Latency:** Higher than block or file storage for small random reads (due to HTTP overhead).
- **Consistency models:** Some object stores offer strong consistency for PUT/DELETE (S3 now has strong consistency), but historically eventual consistency for some operations.
- **Limited operations:** No random writes; you must upload a new version to modify part of an object (though some like Azure Page Blobs support random writes for VHDs).

## 5️⃣ Comparison Table

| Feature | Block Storage | File Storage | Object Storage |
|---------|---------------|--------------|----------------|
| **Access** | Via OS (block device) | Via mounted file system | Via HTTP API |
| **Performance** | Very low latency, high IOPS | Good for throughput, moderate IOPS | Latency higher (HTTP overhead), high throughput for large objects |
| **Scalability** | Limited by volume size (can scale via RAID) | Limited by file system size; cloud file services scale automatically | Virtually unlimited (exabytes) |
| **Sharing** | Single server (or cluster FS) | Multiple servers (NFS/SMB) | Any number of clients via API |
| **Use case** | Databases, OS volumes | Shared files, home directories | Static assets, backups, data lakes |
| **Cost per GB** | High (SSD) / Medium (HDD) | Medium | Low (standard) to very low (archive) |
| **Metadata** | Limited (file system level) | File attributes, extended attributes | Rich custom metadata |
| **Example** | AWS EBS, Azure Disk | AWS EFS, Azure Files | AWS S3, Azure Blob, GCP Storage |

## 6️⃣ Choosing the Right Storage for Your System

In a typical system design, you may use multiple storage types for different purposes. Here’s a decision guide:

### Use Block Storage When:
- You need **low‑latency random I/O** (databases, transactional systems).
- The application expects a raw disk (e.g., runs its own file system).
- You need high IOPS for a single server.
- Example: MySQL database on an EC2 instance with EBS `io2` volumes.

### Use File Storage When:
- Multiple servers need **shared access** to the same files.
- You have a **legacy application** that requires a POSIX file system.
- You need a simple shared workspace (e.g., for user uploads processed by multiple workers).
- Example: A web application where multiple instances serve user‑uploaded images from a shared EFS volume.

### Use Object Storage When:
- You store **large amounts of unstructured data** (images, videos, logs, backups).
- You need **scalability** (exabytes) and **durability** (99.999999999%).
- Data is accessed via **HTTP** (e.g., for web/mobile apps).
- You want **cost‑effective** storage for infrequently accessed data (with lifecycle policies).
- You are building a **data lake** for analytics.
- Example: Storing user profile pictures in S3, served via CloudFront.

### Hybrid Scenarios
- **Database with backup:** Database runs on block storage (EBS); nightly backups are uploaded to object storage (S3).
- **Web application:** Static assets served from object storage via CDN; shared session data stored in Redis (in‑memory); database on block storage.
- **Big data pipeline:** Raw data lands in object storage (data lake); processed by Spark clusters using ephemeral block storage for scratch space; results written back to object storage.

---

## 7️⃣ Key Takeaways for HLD Interviews

- **Block storage** is for low‑latency, single‑server workloads (databases, OS).
- **File storage** is for shared file access across servers.
- **Object storage** is for scalable, durable, HTTP‑accessible storage of unstructured data.
- **Blob storage** is essentially the same as object storage (Azure Blob).
- **Real‑world architectures often combine multiple types** – be ready to explain why you choose each for different parts of the system.

Understanding these storage fundamentals will help you design robust, scalable, and cost‑effective systems for any use case.
