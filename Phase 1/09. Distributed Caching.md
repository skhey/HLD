# Distributed Caching: Consistent Hashing, Cache Replication, and Cache Stampede Prevention

Distributed caching is essential for scaling read-heavy workloads, reducing database load, and providing low‑latency access to frequently used data across multiple servers. In this deep dive, we cover three critical topics that every system designer must understand: **consistent hashing** (how data is distributed across nodes), **cache replication** (how data is copied for fault tolerance), and **cache stampede prevention** (how to avoid cascading failures when a cache entry expires).

---

## 1️⃣ Consistent Hashing

Consistent hashing is a technique used to distribute data across a cluster of cache nodes in a way that minimizes reorganization when nodes are added or removed. It solves the key problem of simple modulo‑based sharding: when the number of nodes changes, most keys need to be remapped, causing massive cache misses and rehashing.

### Why Not Simple Modulo?
In a naive approach, you would assign a key to a node using `hash(key) % N` (where `N` is the number of nodes). If `N` changes (node added or removed), the mapping for **almost all keys** changes, leading to a flood of cache misses and potentially overwhelming the database (cache stampede). Consistent hashing aims to remap only `K/N` keys on average.

### How Consistent Hashing Works
1. **Hash Ring:** Imagine a circular ring of numbers from 0 to 2³²‑1 (or any large range). Each cache node is hashed (using its IP or name) to a point on this ring.
2. **Key Placement:** To find which node holds a given key, you hash the key to a point on the ring and then move clockwise until you encounter the first node. That node is responsible for the key.
3. **Node Addition:** When a new node is added, it is placed on the ring. Only keys that fall between the new node and its predecessor on the ring are remapped to the new node; other keys remain unaffected.
4. **Node Removal:** When a node fails or is removed, its keys are reassigned to the next node clockwise, and no other keys are affected.

```
Ring representation (simplified):

Node A (hash 10)
Node B (hash 50)
Node C (hash 80)
Node D (hash 200)

Key X (hash 30) goes to Node B (next clockwise from 30).
Key Y (hash 210) wraps around to Node A (next after 200 is 10).

Add Node E (hash 40): keys between 40 and 50 move from B to E.
```

### Virtual Nodes (Vnodes)
In practice, a single hash per node can lead to uneven distribution (especially with few nodes). To improve balance, each physical node is represented by multiple **virtual nodes** (replicas) on the ring, each with a different hash. This spreads the load more evenly and also allows heterogeneous weights (nodes with more capacity can have more virtual nodes).

**Example:** A cluster with 3 physical nodes, each with 100 virtual nodes. The ring contains 300 points; keys are evenly spread.

### Advantages
- **Minimal redistribution:** Only `O(K/N)` keys move when nodes join/leave.
- **Scalability:** Adding or removing nodes does not require rehashing all keys.
- **Load balancing:** Virtual nodes help distribute keys uniformly.

### Disadvantages
- **Complexity:** More complex to implement than modulo.
- **Lookup overhead:** Finding the responsible node requires traversing the ring (binary search on sorted list of node hashes), but it's usually O(log N) with tree structures.

### Real‑World Usage
- **Amazon DynamoDB** (and Cassandra) use consistent hashing for data partitioning.
- **Memcached** clients (like libketama) implement consistent hashing.
- **Redis Cluster** uses a variant: each key belongs to a hash slot (0–16383), and slots are assigned to nodes. Slots can be moved between nodes with minimal key migration.

### Code Example (Simplified Python)
```python
import hashlib
import bisect

class ConsistentHash:
    def __init__(self, nodes=None, replicas=100):
        self.replicas = replicas
        self.ring = {}
        self.sorted_keys = []
        if nodes:
            for node in nodes:
                self.add_node(node)

    def _hash(self, key):
        return int(hashlib.md5(key.encode()).hexdigest(), 16)

    def add_node(self, node):
        for i in range(self.replicas):
            virtual_key = f"{node}:{i}"
            hash_val = self._hash(virtual_key)
            self.ring[hash_val] = node
            self.sorted_keys.append(hash_val)
        self.sorted_keys.sort()

    def remove_node(self, node):
        for i in range(self.replicas):
            virtual_key = f"{node}:{i}"
            hash_val = self._hash(virtual_key)
            del self.ring[hash_val]
            self.sorted_keys.remove(hash_val)

    def get_node(self, key):
        if not self.ring:
            return None
        hash_val = self._hash(key)
        # Find the first node with hash >= key_hash
        idx = bisect.bisect(self.sorted_keys, hash_val)
        if idx == len(self.sorted_keys):
            idx = 0  # wrap around
        return self.ring[self.sorted_keys[idx]]

# Usage
ch = ConsistentHash(["node1", "node2", "node3"])
node = ch.get_node("my_key")
```

## 2️⃣ Cache Replication

Replication in distributed caching means maintaining copies of the same cached data on multiple nodes. The primary goals are **high availability** (if one node fails, data is still accessible) and **increased read throughput** (multiple nodes can serve the same data).

### Replication Strategies

#### a) Leader‑Follower (Master‑Slave) Replication
- One node is the **leader** (primary) that handles all writes and updates.
- Followers (secondaries) replicate data asynchronously or synchronously from the leader and serve read requests.
- If the leader fails, a follower can be promoted.

**Examples:** Redis Sentinel (with master‑slave replication), traditional MySQL replication.

**Pros:**
- Simple consistency model: all writes go through the leader, so no write conflicts.
- Reads scale horizontally (followers can be many).

**Cons:**
- Single point of write bottleneck.
- Replication lag can lead to stale reads.

#### b) Multi‑Leader (Multi‑Master) Replication
- Multiple nodes accept writes; each propagates its changes to others.
- Conflict resolution is required (e.g., last‑write‑wins, version vectors, or application‑level merging).

**Examples:** Cassandra (all nodes are equal), Riak, CouchDB.

**Pros:**
- High write availability (any node can accept writes).
- Geographic distribution (writes can happen close to users).

**Cons:**
- Conflict resolution complexity.
- Potential for write amplification.

#### c) Peer‑to‑Peer with Quorum (Dynamo‑style)
- Data is replicated to a configurable number of nodes (replication factor, e.g., 3).
- For a write, the client sends to all replicas but waits for acknowledgment from a **write quorum** (e.g., 2 out of 3). Reads similarly require a **read quorum**.
- This provides tunable consistency (e.g., `QUORUM` read + write ensures strong consistency if `R + W > N`).

**Examples:** Cassandra, DynamoDB (underlying).

### Consistency vs. Availability in Replication
- **Synchronous replication:** Write waits for all replicas to confirm – strong consistency but higher latency and lower availability if a replica is down.
- **Asynchronous replication:** Write acknowledges immediately – low latency, but risk of data loss if the primary fails before replicating.

### Cache‑Specific Considerations
- **TTL and invalidation:** Replicated caches must handle expiration consistently. If one node expires an entry while another still has it, stale reads can occur. Solutions: use a global TTL, or publish invalidation messages.
- **Replication lag:** In a leader‑follower setup, a read from a follower might miss a recent write. For read‑after‑write consistency, you may need to read from the leader or use session stickiness.

### Real‑World Examples
- **Redis Cluster:** Data is sharded across masters, each master can have one or more replicas. Replication is asynchronous. Clients can read from replicas (using `READONLY` mode) to scale reads.
- **Memcached:** Does not natively support replication; it is a pure distributed cache with no cross‑node communication. Replication must be implemented at the client level (e.g., write to multiple Memcached servers).
- **Cassandra:** All nodes are equal; data is replicated to a configurable number of nodes based on the replication factor and partitioner (consistent hashing).

## 3️⃣ Cache Stampede Prevention

A **cache stampede** (also called a **thundering herd** or **cache miss storm**) occurs when a popular cache entry expires or is invalidated, and multiple concurrent requests all attempt to recompute the expensive value simultaneously. This can overwhelm the backend (database, compute service) and cause cascading failures.

### The Problem Scenario
1. A cached value with TTL = 60 seconds expires.
2. At that moment, 1000 requests arrive for that key.
3. All 1000 requests see a cache miss and simultaneously hit the database to recompute the value.
4. The database becomes overloaded, leading to timeouts or crashes.
5. Subsequent requests may also fail, causing a system‑wide outage.

### Prevention Techniques

#### a) Early Recomputation (Proactive Refresh)
Before the value expires, proactively refresh it in the background. This can be done by a background job that periodically recomputes and updates the cache, or by the cache itself when it detects that the entry is nearing expiration.

**Implementation:**
- Set a TTL shorter than the actual desired expiration, and have a background worker refresh the value before it expires.
- In systems like Redis, you can use keyspace notifications to trigger a refresh.

**Pros:**
- Smooths load; no sudden spikes.

**Cons:**
- Requires a background process; may recompute unnecessarily if the value is rarely accessed.

#### b) Probabilistic Early Expiration (X‑Fetch)
When a request retrieves a cached value, also check how long until expiration. If the remaining TTL is below a threshold, probabilistically decide to recompute early. This spreads the recomputation load over time.

**Example (from Facebook's McDipper):**
```python
def get_with_probabilistic_refresh(key):
    value, ttl_remaining = cache.get_with_ttl(key)
    if ttl_remaining < EARLY_RECALC_THRESHOLD:
        if random.random() < (EARLY_RECALC_THRESHOLD - ttl_remaining) / EARLY_RECALC_THRESHOLD:
            # Refresh in background (async)
            async_refresh(key)
    return value
```

**Pros:**
- No background daemon needed; load is naturally distributed.

**Cons:**
- Slightly more complex; requires tracking TTL.

#### c) Mutex / Locking
When a cache miss occurs, only one request is allowed to recompute the value; others wait and then read the newly cached value.

**Implementation (pseudo‑code with Redis locks):**
```python
def get_with_lock(key):
    value = cache.get(key)
    if value is not None:
        return value

    # Try to acquire a lock for this key
    lock_key = f"lock:{key}"
    if cache.setnx(lock_key, "locked", expire=10):  # acquire lock
        try:
            # Double‑check cache (another process might have updated it)
            value = cache.get(key)
            if value is None:
                value = expensive_computation()
                cache.set(key, value, ttl=60)
            return value
        finally:
            cache.delete(lock_key)
    else:
        # Wait for the lock to be released (poll with backoff)
        sleep(0.1)
        return get_with_lock(key)  # recursive retry
```

**Pros:**
- Guarantees that only one request recomputes.
- Works well for moderate contention.

**Cons:**
- Adds latency for waiting requests.
- Risk of deadlock if lock is not released properly (use TTL on lock).

#### d) External Computation (Background Refresh with Queue)
Instead of recomputing in the request path, send a message to a queue to recompute the value asynchronously, and serve a slightly stale value in the meantime.

**Implementation:**
- On cache miss, serve a stale value (if available) and queue a refresh job.
- The refresh job updates the cache.

**Pros:**
- Zero impact on request latency.
- Gracefully handles high load.

**Cons:**
- Serves stale data; may not be acceptable for all use cases.
- Requires queue infrastructure.

#### e) Cache‑Aside with Short TTL + Retry
Combine a short TTL with retry logic. This reduces the window of concurrent misses.

**Example:** Set TTL to 5 seconds. When a miss occurs, recompute, but also implement a small random delay before recomputing (jitter) to spread the load.

### Comparison

| Technique | Complexity | Staleness Risk | Additional Infrastructure | Use Case |
|-----------|------------|----------------|---------------------------|----------|
| Early Recomputation | Medium | None | Background workers | Predictable access patterns |
| Probabilistic Early Expiration | Medium | None | None (client‑side) | High‑traffic, random access |
| Mutex / Lock | Low | None | Distributed lock | Moderate contention |
| Background Queue | High | Stale allowed | Message queue | Write‑heavy, stale‑tolerant |
| Short TTL + Jitter | Low | Stale minimal | None | Simple, low‑criticality |

### Real‑World Examples
- **Redis** does not natively prevent stampedes; you must implement one of the above in your application.
- **Memcached** also requires client‑side stampede protection.
- **Facebook** uses a combination of lease gets (a form of locking) and background refresh in their Memcached deployment.
- **Google's Memcache** service uses a token‑based system to limit concurrent recomputations.

## 4️⃣ Summary and Key Takeaways

| Topic | Key Points |
|-------|------------|
| **Consistent Hashing** | Distributes keys across nodes with minimal remapping on topology changes. Uses a ring and virtual nodes for even load. Essential for scalable distributed caches (Cassandra, DynamoDB, Redis Cluster). |
| **Cache Replication** | Provides fault tolerance and read scalability. Strategies: leader‑follower (simpler, but write bottleneck), multi‑leader (high write availability, conflict resolution), quorum‑based (tunable consistency). |
| **Cache Stampede Prevention** | Avoids thundering herds when popular entries expire. Techniques: early recompute, probabilistic expiration, locking, background queues. Choose based on consistency needs and load. |

Understanding these concepts allows you to design robust distributed caching layers that can handle large‑scale traffic, node failures, and peak loads without collapsing the backend.
