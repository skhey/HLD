# Caching Fundamentals: Strategies and Eviction Policies

Caching is a critical technique in high‑level design to improve system performance, reduce latency, and offload backend services. By storing frequently accessed data in a fast, temporary storage layer (the cache), you can serve requests much faster than fetching from the primary database. This guide covers the core caching strategies (cache‑aside, read/write‑through, write‑back, write‑around) and the most common eviction policies (LRU, LFU, TTL) with their trade‑offs and real‑world examples.

---

## 1️⃣ Why Cache?

- **Reduce latency:** Serving data from memory (microseconds) is orders of magnitude faster than from disk (milliseconds).
- **Decrease database load:** Fewer queries to the database mean lower CPU and I/O usage, allowing it to handle more writes or complex queries.
- **Improve throughput:** Caches can handle a high volume of read requests, increasing overall system capacity.
- **Handle traffic spikes:** A well‑cached system can absorb sudden increases in read traffic without collapsing the database.

### Where Caching Is Applied
- **Client‑side:** Browser cache, mobile app cache.
- **CDN:** Edge caches for static assets.
- **Reverse proxy / API gateway:** Nginx, Varnish.
- **Application‑level:** In‑memory caches like Redis, Memcached.
- **Database‑level:** Buffer pools, query caches (though often discouraged).

## 2️⃣ Caching Strategies

The caching strategy defines how data is read from and written to the cache in relation to the underlying database. The choice affects consistency, performance, and complexity.

### 2.1 Cache‑Aside (Lazy Loading)
**How it works:**
- The application is responsible for interacting with both the cache and the database.
- On a read request:
  1. Check the cache for the data.
  2. If found (cache hit), return it.
  3. If not found (cache miss), load the data from the database, store it in the cache, and return it.
- On a write request:
  1. Write directly to the database.
  2. Optionally invalidate or update the cache entry (or let it expire).

```
Read:    Application → Cache (miss) → Database → Cache (set) → Client
Write:   Application → Database → (optional cache update)
```

**Pros:**
- Cache only contains data that is actually requested (efficient memory usage).
- Simple to implement.
- Resilient to cache failures – the system can still fall back to the database.

**Cons:**
- Cache misses add latency (three steps instead of one).
- Data can become stale if the database is updated but the cache is not invalidated.
- Requires application logic to handle cache misses and updates.

**When to use:** General‑purpose workloads where read‑to‑write ratio is high and some staleness is acceptable.

**Example (Pseudocode):**
```python
def get_user(user_id):
    # Check cache
    user = cache.get(f"user:{user_id}")
    if user:
        return user
    # Cache miss – load from DB
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    cache.set(f"user:{user_id}", user, ttl=3600)
    return user
```

### 2.2 Read‑Through
**How it works:**
- The cache sits between the application and the database.
- The application always reads from the cache.
- On a cache miss, the cache itself loads the missing data from the database, stores it, and returns it to the application.

```
Read:    Application → Cache (miss) → Cache loads from DB → Cache stores → Client
Write:   (depends on write strategy – often write‑through or write‑around)
```

**Pros:**
- Simplifies application code – the application only talks to the cache.
- The cache can coordinate loading and refreshing data (e.g., using read‑ahead).

**Cons:**
- Requires the cache to have logic to load from the database (not all caches support this; e.g., Redis doesn't natively, but you can implement it in the client library).
- Still has cache miss penalty.

**When to use:** When you want to abstract the database away from the application, or when using caching libraries like Hibernate/ORM that support read‑through.

**Example with a hypothetical cache library:**
```java
// Configuration: cache loader knows how to fetch from DB
CacheLoader<String, User> loader = key -> db.loadUserById(key);
LoadingCache<String, User> cache = Caffeine.newBuilder()
    .build(loader);
// Application code
User user = cache.get("user:123"); // automatically loads on miss
```

### 2.3 Write‑Through
**How it works:**
- Writes are first performed on the cache, and then the cache synchronously writes the data to the database.
- The write is considered successful only after both cache and database updates are complete.

```
Write:   Application → Cache → Database → (both updated) → Client
Read:    (can be any read strategy, often read‑through)
```

**Pros:**
- Ensures cache is always consistent with the database (no stale data).
- Application code is simple – just write to the cache.

**Cons:**
- Higher write latency (must wait for both cache and DB).
- If the database is down, writes fail.
- Cache may become polluted with rarely read data (if writes are frequent but reads are not).

**When to use:** Systems where read‑after‑write consistency is critical and write latency is acceptable.

**Example:**
```python
def update_user(user_id, data):
    # Update cache
    cache.set(f"user:{user_id}", data)
    # Synchronously update database
    db.execute("UPDATE users SET ... WHERE id = ?", data)
```

### 2.4 Write‑Back (Write‑Behind)
**How it works:**
- Writes are written to the cache only and immediately acknowledged.
- The cache asynchronously writes the data to the database after a delay or in batches.

```
Write:   Application → Cache (ack) → (later) Cache → Database
Read:    Always from cache (which has the latest data)
```

**Pros:**
- Very low write latency (only cache write).
- Can batch writes to the database, reducing load.
- Tolerates database outages (writes are queued).

**Cons:**
- Risk of data loss if the cache fails before persisting to the database.
- Complexity – need to handle write failures, retries, and ordering.
- Consistency window where database is stale.

**When to use:** Write‑heavy workloads where some data loss is acceptable (e.g., analytics, counters, logs). Often combined with a durable message queue.

**Example (with a write‑behind cache library):**
```java
Cache<String, User> cache = Caffeine.newBuilder()
    .writer(new CacheWriter<String, User>() {
        @Override
        public void write(String key, User user) {
            // Asynchronously write to DB (e.g., send to Kafka)
            kafkaProducer.send(new Record(key, user));
        }
    })
    .build();
cache.put("user:123", updatedUser); // returns immediately
```

### 2.5 Write‑Around
**How it works:**
- Writes go directly to the database, bypassing the cache.
- The cache is updated only on a subsequent read (cache‑aside style).

```
Write:   Application → Database (cache untouched)
Read:    Cache → (if miss) → Database → populate cache
```

**Pros:**
- Cache is not polluted by data that is written but rarely read.
- Simple and efficient for write‑heavy, read‑rare workloads.

**Cons:**
- Reads after a write will incur a cache miss (higher latency until the cache is populated).
- Cache may be stale if the same data is later updated and not invalidated.

**When to use:** When data is written infrequently and read often, or when you want to avoid filling the cache with transient data (e.g., logs).

### Summary of Write Strategies

| Strategy        | Write Latency | Read Latency | Data Freshness | Use Case |
|-----------------|---------------|--------------|----------------|----------|
| **Write‑Around**| DB only       | Miss on first read after write | Stale until read | Write‑once, read‑rarely |
| **Write‑Through**| High (sync)   | Low (always cached) | Always consistent | Critical data, low write volume |
| **Write‑Back**  | Very low      | Low (cached) | Eventual consistency | High write throughput, loss‑tolerant |

## 3️⃣ Eviction Policies

Caches have limited memory. When the cache is full, some entries must be removed to make room for new ones. The eviction policy determines which entries are removed. The most common policies are:

### 3.1 LRU (Least Recently Used)
**How it works:** Evicts the item that has not been accessed for the longest time. Based on the assumption that items used recently are likely to be used again soon (temporal locality).

**Implementation:** Typically a combination of a hash map (for O(1) lookup) and a doubly linked list (to maintain access order). On each access, the item is moved to the front of the list. Eviction removes from the tail.

**Pros:**
- Simple and effective for many workloads.
- Good temporal locality.

**Cons:**
- Can be fooled by one‑time large scans (pollutes cache).
- Overhead of maintaining access order.

**Used by:** Redis (when `maxmemory-policy` is set to `allkeys-lru`), Memcached (default), CPU caches.

### 3.2 LFU (Least Frequently Used)
**How it works:** Evicts the item with the smallest access frequency. Keeps a counter per item and evicts the one with the lowest count. Some variants use aging to prevent old counters from dominating.

**Pros:**
- Retains frequently accessed items, even if not recently used.
- Good for workloads with stable popularity (e.g., product catalog).

**Cons:**
- More complex to implement (needs frequency counters).
- Can suffer from “pollution” by past popularity that has declined (aging helps).
- Overhead of maintaining counters.

**Used by:** Redis (with `volatile-lfu` or `allkeys-lfu`), some custom caches.

### 3.3 TTL (Time‑To‑Live)
**How it works:** Each cache entry has an expiration time. After that time, the entry is automatically removed. This is not strictly an eviction policy for capacity, but a data freshness mechanism. However, when combined with LRU/LFU, TTL can help remove stale data.

**Pros:**
- Ensures data does not become too stale.
- Simple – set a fixed expiration.

**Cons:**
- Does not consider memory usage; if many items expire at once, it can cause a spike in cache misses.
- Requires application to set appropriate TTL.

**Used by:** Redis (`EXPIRE`), Memcached (max TTL 30 days), most caching libraries.

### 3.4 Other Policies (Briefly)

| Policy | Description | Use Case |
|--------|-------------|----------|
| **FIFO (First In, First Out)** | Evicts the oldest item by insertion time. | Simple, but rarely used due to poor performance. |
| **MRU (Most Recently Used)** | Evicts the most recently used item. | Opposite of LRU; used in some specific patterns (e.g., caching where older data is more valuable). |
| **Random** | Randomly picks an item to evict. | Simple, low overhead; surprisingly effective in some workloads. |
| **LFU with Aging** | LFU where frequency counters decay over time to favor recent popularity. | Better adaptation to changing access patterns. |

### Choosing an Eviction Policy

| Workload Characteristic | Recommended Policy |
|-------------------------|---------------------|
| Strong temporal locality (e.g., user sessions) | LRU |
| Stable popularity distribution (e.g., product catalog) | LFU |
| Data must be fresh (e.g., stock prices) | TTL (with any policy) |
| Mixed, unpredictable | LRU + TTL (or LFU + TTL) |
| Minimal overhead, low hit rate acceptable | Random |

### Redis Example: Configuring Eviction
In `redis.conf` or via `CONFIG SET`:
```
maxmemory 2gb
maxmemory-policy allkeys-lru   # or allkeys-lfu, volatile-lru, etc.
maxmemory-samples 5            # LRU/LFU approximated via sampling for performance
```

## 4️⃣ Caching Strategies + Eviction: Putting It Together

In practice, you combine a caching strategy with an eviction policy. For example:
- **Cache‑aside + LRU:** The most common combination. Application manages cache misses and writes; cache evicts least recently used items when full.
- **Read‑through + TTL:** Cache loads on miss; entries expire after a fixed time, forcing a reload.
- **Write‑back + LFU:** Writes are batched; frequently accessed items stay in cache.

### Example: Redis as Cache‑Aside with LRU + TTL
```python
# Set with expiration (TTL)
redis.setex("user:123", 3600, serialized_user)

# On read
user_data = redis.get("user:123")
if not user_data:
    user_data = db_load_user(123)
    redis.setex("user:123", 3600, user_data)
```

If Redis memory fills, it evicts using the configured policy (e.g., `allkeys-lru`).
