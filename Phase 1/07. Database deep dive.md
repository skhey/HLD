# Database Deep Dive: SQL vs. NoSQL, Read Replicas, and Partitioning Strategies

This comprehensive guide covers three critical database topics that frequently appear in system design interviews and real‑world architecture decisions. You'll learn how to choose between SQL and NoSQL, how to scale reads using read replicas, and how to distribute data across multiple nodes with partitioning.

---

## 1️⃣ Choosing Between SQL and NoSQL

The decision between a relational (SQL) and a non‑relational (NoSQL) database is foundational. It impacts your system's scalability, consistency, and development velocity. There is no one‑size‑fits‑all answer; the choice depends on your specific requirements.

### Key Decision Factors

| Factor | SQL | NoSQL |
|--------|-----|-------|
| **Data Structure** | Structured, fixed schema with relationships (normalized). | Semi‑structured or unstructured; flexible schema (denormalized). |
| **Schema Flexibility** | Rigid; migrations require downtime or careful planning. | Dynamic; fields can vary per record, allowing rapid iteration. |
| **Consistency** | Strong consistency (ACID) by default. | Typically eventual consistency (BASE); some offer tunable consistency. |
| **Scalability** | Vertical scaling (scale‑up) is common; horizontal scaling (sharding) is possible but complex. | Designed for horizontal scaling (scale‑out) from the start. |
| **Query Complexity** | Powerful, expressive queries with JOINs, aggregations, subqueries. | Limited query capabilities; joins are either unsupported or inefficient. |
| **Use Cases** | Financial systems, inventory, ERP, applications with complex relationships. | Real‑time analytics, content management, IoT, caching, social networks. |
| **Development Velocity** | Slower changes due to schema rigidity; strong data integrity. | Faster iterations; can start coding without upfront schema design. |
| **Team Expertise** | Widely understood; many developers know SQL. | Requires learning new data models and query languages. |

### Detailed Comparison Table

| Feature | SQL (Relational) | NoSQL |
|---------|------------------|-------|
| **Data Model** | Tables with rows and columns, fixed schema. | Key‑Value, Document, Wide‑Column, Graph. |
| **ACID Transactions** | Fully supported. | Limited or no multi‑record ACID (exceptions: MongoDB 4.0+, some graph DBs). |
| **Joins** | Natural, optimized. | Not supported (except graph DBs); denormalization required. |
| **Indexing** | Rich indexing (B‑Tree, hash, bitmap, full‑text). | Varies by type: secondary indexes in document stores, limited in key‑value. |
| **Replication** | Master‑slave, sometimes multi‑master. | Often masterless (peer‑to‑peer) with tunable consistency. |
| **Partitioning** | Manual sharding (application‑level) or via middleware. | Built‑in sharding (e.g., Cassandra, MongoDB). |
| **Maturity** | Decades of maturity, tooling, and community. | Newer, but many have proven production use (Cassandra, MongoDB, Redis). |

### Decision Flowchart (Mental Model)

```
Start
│
├─ Do you need ACID transactions across multiple records? → Yes → SQL
│
├─ Is your data highly structured with complex relationships and joins? → Yes → SQL
│
├─ Does your schema change frequently (agile development)? → Yes → Consider NoSQL (Document)
│
├─ Do you need massive write scalability (millions of writes/sec)? → Yes → Consider NoSQL (Wide‑Column or Key‑Value)
│
├─ Is your access pattern simple (key‑value lookups)? → Yes → Consider NoSQL (Key‑Value)
│
├─ Are you dealing with highly connected data (e.g., social graphs)? → Yes → Consider Graph NoSQL
│
└─ Otherwise, SQL is often a safe default, especially for business‑critical data.
```

### Example Scenarios

- **Banking System:** SQL (ACID, complex transactions, strong consistency).
- **E‑commerce Platform:** Hybrid – SQL for orders/inventory (ACID), NoSQL for product catalog (flexible schema) and user sessions (key‑value).
- **IoT Sensor Data:** Wide‑column NoSQL (e.g., Cassandra) for high write throughput and time‑series queries.
- **Social Network:** Graph database for friend relationships and recommendations; document store for user posts.
- **Real‑Time Leaderboard:** Redis (key‑value) for fast updates and sorted sets.

## 2️⃣ Read Replicas

Read replicas are copies of your primary database that serve read‑only queries. They are a common pattern to scale read throughput and improve availability.

### How Read Replicas Work

- **Replication:** The primary database (master) continuously streams its write operations (via binary log, write‑ahead log) to replica servers.
- **Asynchronous vs. Synchronous:**
  - **Asynchronous (most common):** Primary commits writes without waiting for replicas. Fast, but replicas may lag.
  - **Synchronous:** Primary waits for at least one replica to confirm the write. Ensures zero data loss but increases write latency.
- **Application Integration:** The application directs write queries to the primary and read queries to replicas (often via a load balancer or middleware).

```
        ┌─────────────────┐
        │   Application   │
        └────────┬────────┘
                 │
       ┌─────────┴─────────┐
       │                   │ (read)
       ▼                   ▼
┌──────────────┐    ┌──────────────┐
│   Primary     │ ←  │   Replica 1  │
│   (Master)    │    │   (Slave)    │
└──────────────┘    └──────────────┘
       │ replication        │
       └────────────────────┘
```

### Benefits

- **Read Scalability:** Offload read traffic from the primary, allowing it to focus on writes.
- **High Availability:** If the primary fails, a replica can be promoted (failover).
- **Reporting/Analytics:** Run heavy analytical queries on replicas without impacting production performance.
- **Geographic Distribution:** Place replicas closer to users to reduce read latency.

### Drawbacks and Challenges

- **Replication Lag:** Asynchronous replicas may be behind the primary. A read from a replica might return stale data.
- **Consistency Concerns:** If your application requires read‑after‑write consistency (user sees their own changes immediately), you must route those reads to the primary or wait for replication.
- **Failover Complexity:** Promoting a replica requires handling potential data loss (if replication was async) and updating application connection strings.
- **Additional Cost:** Running replica servers incurs infrastructure and operational costs.

### Handling Read‑After‑Write Consistency

- **Read‑Your‑Writes:** After a user performs a write, send their subsequent reads to the primary for a short period.
- **Version Stamps:** Include a timestamp or version in the response; the client includes it in reads; replicas return data only if they are at least that version.
- **Synchronous Replication for Critical Data:** Ensure some replicas are synchronously updated before acknowledging the write (increases latency but guarantees consistency).

### Example: Using Read Replicas in a Web Application

```python
# Pseudocode for routing reads
def get_user_profile(user_id, after_write=False):
    if after_write:
        # User just updated their profile; read from primary
        db = primary_connection
    else:
        # Normal read; use replica
        db = replica_connection
    return db.query("SELECT * FROM users WHERE id = ?", user_id)
```

### Real‑World Examples

- **Amazon RDS:** Supports creating up to 5 read replicas (MySQL, PostgreSQL) with automatic replication.
- **Azure SQL Database:** Active geo‑replication for read‑scale and disaster recovery.
- **Cassandra:** All nodes are equal; you can route reads to any node, but consistency levels (ONE, QUORUM, ALL) control how many replicas must respond.

## 3️⃣ Partitioning Strategies

Partitioning (also called sharding in distributed systems) is the technique of splitting a large database into smaller, independent parts to improve scalability, manageability, and performance.

### Why Partition?

- **Scalability:** Distribute data across multiple servers to handle more writes and storage than a single machine.
- **Performance:** Queries that can be routed to a single partition are faster because they scan less data.
- **Availability:** Failures are isolated to a subset of partitions; the rest of the system remains available.
- **Manageability:** Smaller databases are easier to backup, restore, and maintain.

### Types of Partitioning

#### a) Horizontal Partitioning (Sharding)
Splits rows of a table across multiple databases. Each shard has the same schema but holds a different subset of data. This is the most common form of partitioning for scalability.

##### Sharding Strategies

| Strategy | Description | Pros | Cons |
|----------|-------------|------|------|
| **Range‑Based** | Partition by a contiguous range of the shard key (e.g., user_id 1–10k in shard 0, 10k–20k in shard 1). | Simple to implement; supports range queries on the shard key. | Can cause hot spots if distribution is uneven (e.g., recent users in one shard). |
| **Hash‑Based** | Apply a hash function to the shard key, then take modulo (e.g., `hash(user_id) % N`). | Even distribution of data; avoids hot spots. | Adding/removing shards requires rehashing (data movement). |
| **Directory‑Based** | Maintain a lookup table that maps each key to its shard. | Flexible; easy to rebalance; allows arbitrary mapping. | Lookup table becomes a bottleneck and a single point of failure (unless distributed). |

**Example: Hash‑Based Sharding of Users Table**
```sql
-- Number of shards = 4
-- shard_id = user_id % 4
-- Shard 0: user_id % 4 == 0
-- Shard 1: user_id % 4 == 1, etc.
```

#### b) Vertical Partitioning
Splits a table by **columns** into multiple tables, often to separate frequently accessed columns from large, less‑used ones (e.g., moving BLOBs to a separate table). It can also mean splitting the database schema into different tables that reside on different servers, but usually vertical partitioning is done within a single database.

**Example:** Split `users` table into `user_profile` (frequently accessed: id, name, email) and `user_preferences` (less frequent: theme, notifications).

#### c) Functional Partitioning (Microservices)
Partition by **business domain** – each microservice owns its own database. This is not strictly a database technique but an architectural choice that naturally partitions data.

### Choosing a Shard Key

The shard key is the most critical decision in sharding. It determines how data is distributed and how efficiently queries can be routed.

**Criteria for a Good Shard Key:**

- **High Cardinality:** Many distinct values to distribute evenly.
- **Even Access Pattern:** Writes and reads should be spread across shards.
- **Query‑Friendly:** Most queries should include the shard key to avoid scatter‑gather (querying all shards).
- **Immutable or Rarely Changing:** Changing the shard key after data is written requires expensive data migration.

**Examples:**
- `user_id` for a users table (good cardinality, queries often filter by user_id).
- `order_id` for orders (similarly good).
- `timestamp` for time‑series data (but may cause hot shard if recent data goes to one node – use bucket sharding: `hash(timestamp_bucket)`).

### Consistent Hashing

A special hashing technique that minimizes rehashing when the number of shards changes. Each shard is assigned a range of hash values (points on a circle). Data is placed on the first shard encountered clockwise from its hash. When adding a shard, only data from the immediate neighbor is redistributed.

**Used by:** DynamoDB, Cassandra, Riak.

### Challenges of Sharding

- **Cross‑Shard Queries:** Queries that need data from multiple shards are complex and slow. You may need to query all shards and aggregate results in the application.
- **Joins and Transactions:** ACID transactions across shards are extremely hard. Design your system to avoid cross‑shard operations, or use distributed transaction protocols (e.g., two‑phase commit) with performance penalties.
- **Rebalancing:** When adding or removing shards, data must be moved. This is a complex, often manual process (though some databases like Cassandra rebalance automatically).
- **Data Skew:** Some shards may become hot if distribution isn't even (e.g., celebrity users in a social network). Mitigations: use a more granular shard key, split hot shards, or use caching.
- **Backup and Restore:** Must coordinate across all shards; consistency may be tricky.

### Example: Sharding an E‑Commerce Database

- **Users table:** Sharded by `user_id` (hash modulo) – a user's data stays on one shard.
- **Orders table:** Also sharded by `user_id` to keep all orders of a user on the same shard (enables local joins). Alternatively, shard by `order_id` if orders are accessed individually.
- **Product catalog:** Not sharded (or replicated globally) because it's read‑heavy and relatively small.
- **Inventory:** Could be sharded by `product_id` or kept separate with careful consistency handling.

## 4️⃣ Summary and Key Takeaways for Interviews

| Topic | Key Points |
|-------|------------|
| **SQL vs. NoSQL** | Choose SQL for ACID, complex queries, structured data; NoSQL for scale, flexibility, and specific access patterns. Always justify with requirements. |
| **Read Replicas** | Use to scale reads, improve availability, and offload analytics. Be aware of replication lag and design for eventual consistency when appropriate. |
| **Sharding** | Horizontal partitioning is essential for write scalability. Pick a high‑cardinality, evenly‑distributed shard key. Understand trade‑offs: cross‑shard queries vs. local queries, rebalancing, and data skew. |

In system design interviews, you'll often be asked to design a data layer. Be prepared to discuss these concepts, compare alternatives, and explain the trade‑offs based on the system's functional and non‑functional requirements.

**Example Discussion Snippet:**
> "Given the need for high write throughput (millions of events per day), we will use Cassandra, a wide‑column NoSQL database, because it scales horizontally and offers tunable consistency. We'll partition the events table by `user_id` and cluster by `timestamp` to allow efficient time‑range queries per user. For read scalability, we'll rely on Cassandra's built‑in replication; all nodes can serve reads. We won't use read replicas in the traditional sense because Cassandra's architecture already distributes reads."

Mastering these concepts will enable you to design robust, scalable data architectures that can grow with your application.
