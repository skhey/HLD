# Reliability Patterns: Circuit Breaker, Retry with Backoff, Bulkhead, Health Checks, and Rate Limiting

Building resilient distributed systems requires anticipating and handling failures gracefully. These five reliability patterns are essential tools in the architect's toolkit. They help prevent cascading failures, maintain system stability under load, and ensure that your application can recover from transient issues.

---

## 1️⃣ Circuit Breaker

The Circuit Breaker pattern prevents an application from repeatedly trying to execute an operation that is likely to fail. It monitors for failures and, once a threshold is reached, it "trips" and subsequent calls fail immediately without attempting the operation. This gives the failing component time to recover.

### States
- **Closed:** Normal operation. Requests pass through. Failures are counted. If failures exceed a threshold within a time window, the breaker trips to **Open**.
- **Open:** Requests fail immediately (without attempting the operation). After a timeout, the breaker transitions to **Half‑Open**.
- **Half‑Open:** A limited number of test requests are allowed. If they succeed, the breaker resets to **Closed**; if they fail, it goes back to **Open**.

### Why Use It?
- Prevents overloading a failing service.
- Gives the downstream system time to recover.
- Failing fast avoids wasted resources and long timeouts.

### Implementation Considerations
- Failure threshold (e.g., 5 failures in 10 seconds).
- Timeout before transitioning from Open to Half‑Open.
- Test request count in Half‑Open state.
- Which exceptions count as failures (e.g., timeouts, 5xx HTTP status codes).

### Example (Pseudocode with Resilience4j‑style)
```java
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(50) // 50% failures
    .slidingWindowSize(10)
    .waitDurationInOpenState(Duration.ofSeconds(5))
    .permittedNumberOfCallsInHalfOpenState(3)
    .build();

CircuitBreaker circuitBreaker = CircuitBreaker.of("backendService", config);

Supplier<String> decorated = CircuitBreaker.decorateSupplier(circuitBreaker, 
    () -> backendService.call());

Try<String> result = Try.ofSupplier(decorated)
    .recover(throwable -> "fallback response");
```

### Real‑World Examples
- Netflix Hystrix (now in maintenance mode), Resilience4j.
- AWS SDK retries and circuit breaker in clients.

## 2️⃣ Retry with Backoff

Transient failures (e.g., network glitches, temporary unavailability) can often be resolved by retrying the operation after a short delay. Retry with backoff is a strategy that retries failed operations with increasing delays between attempts.

### Why Backoff?
- Without backoff, retries can overwhelm the recovering service.
- Exponential backoff (and jitter) spreads out retries and reduces contention.

### Backoff Strategies
- **Fixed delay:** Wait a constant time between retries. Simple but may still cause thundering herd.
- **Exponential backoff:** Delay increases exponentially (e.g., 100ms, 200ms, 400ms, ...). Reduces load.
- **Exponential backoff with jitter:** Add randomness to the delay (e.g., random between 0 and current backoff) to prevent synchronized retries.

### Implementation Considerations
- Maximum retries.
- Which errors are retryable (e.g., 5xx, network timeouts) vs. not (4xx client errors).
- Idempotency – ensure retries do not cause duplicate side effects.
- Backoff formula example: `delay = min(cap, base * 2^retry) + random_jitter`.

### Example (Exponential Backoff with Jitter in Python)
```python
import time
import random

def call_with_retry(func, max_retries=5, base_delay=0.1):
    for attempt in range(max_retries):
        try:
            return func()
        except RetryableError as e:
            if attempt == max_retries - 1:
                raise e
            delay = base_delay * (2 ** attempt) + random.uniform(0, 0.1 * (2 ** attempt))
            time.sleep(delay)
    # unreachable
```

### Real‑World Examples
- AWS SDKs: automatic retries with exponential backoff.
- Database connection libraries often retry transient errors.

## 3️⃣ Bulkhead

The Bulkhead pattern isolates different parts of a system so that a failure in one component does not cascade and bring down others. Inspired by ship bulkheads – if one compartment floods, the ship stays afloat.

### Concept
- Limit the resources (threads, connections, memory) allocated to each service or client.
- If one service becomes overloaded or fails, it only exhausts its own pool, leaving others unaffected.

### Implementation Approaches
- **Thread pools:** Assign separate thread pools for different dependencies. If one pool is exhausted, other dependencies still have available threads.
- **Semaphores (concurrency limits):** Limit the number of concurrent calls to a dependency.
- **Connection pools:** Separate connection pools for different databases/services.
- **Queues with bounded capacity:** Use bounded queues for work destined for a particular service.

### Example (Java with Thread Pools)
```java
ExecutorService dependencyAPool = Executors.newFixedThreadPool(10);
ExecutorService dependencyBPool = Executors.newFixedThreadPool(10);

Future<String> resultA = dependencyAPool.submit(() -> callServiceA());
Future<String> resultB = dependencyBPool.submit(() -> callServiceB());
```

### Real‑World Examples
- Hystrix thread‑pool isolation.
- Resilience4j bulkhead (semaphore or thread pool).
- Kubernetes resource limits (CPU/memory) also serve as bulkheads at the infrastructure level.

## 4️⃣ Health Checks

Health checks are endpoints or procedures that report the status of a service or component. They are used by load balancers, orchestrators (Kubernetes), and monitoring systems to determine if a service is alive and ready to receive traffic.

### Types of Health Checks
- **Liveness:** Indicates whether the service is running (not crashed or deadlocked). If liveness fails, the service may be restarted.
- **Readiness:** Indicates whether the service is ready to accept traffic (e.g., it has finished startup, dependencies are available). If readiness fails, the service is taken out of the load balancer rotation.

### Implementation
- Expose an HTTP endpoint like `/health` or `/ready`.
- The endpoint can perform internal checks (database connectivity, disk space, etc.) and return HTTP 200 if healthy, 5xx otherwise.
- Lightweight checks should be fast and not overload the service.

### Example (Express.js)
```javascript
app.get('/health', (req, res) => {
  if (database.isConnected() && cache.isConnected()) {
    res.status(200).send('OK');
  } else {
    res.status(503).send('Service Unavailable');
  }
});
```

### In Kubernetes
- **livenessProbe:** Periodically checks if container is alive; restarts if fails.
- **readinessProbe:** Checks if container is ready to serve traffic; removes from service endpoints if fails.

### Real‑World Examples
- AWS ELB health checks (ping a path, e.g., `/index.html`).
- Consul health checks for service discovery.
- Spring Boot Actuator health endpoint.

## 5️⃣ Rate Limiting

Rate limiting controls the rate of traffic sent or received by a service. It protects the system from being overwhelmed by too many requests, whether from a malicious actor, a buggy client, or a legitimate traffic spike.

### Why Rate Limit?
- Prevent resource exhaustion (CPU, memory, database connections).
- Ensure fair usage among tenants (in multi‑tenant systems).
- Throttle API clients to stay within capacity.
- Mitigate DoS attacks.

### Common Algorithms

#### a) Token Bucket
- A bucket holds tokens; tokens are added at a fixed rate.
- Each request consumes a token; if tokens are available, the request proceeds; otherwise, it is delayed or rejected.
- Allows bursts up to bucket size.

#### b) Leaky Bucket
- Requests enter a queue (bucket) and are processed at a fixed rate.
- If the queue is full, new requests are rejected.
- Smooths output, no bursts.

#### c) Fixed Window Counter
- Count requests per fixed time window (e.g., 1 minute). When the counter exceeds the limit, reject.
- Simple but can allow double the limit at window boundaries.

#### d) Sliding Window Log
- Tracks timestamps of requests within a sliding window.
- More accurate but memory‑intensive.

#### e) Sliding Window Counter
- Hybrid: divides time into small buckets, approximates sliding window.

### Implementation Considerations
- Where to enforce: client‑side, gateway, or application.
- Rate limit key: IP address, user ID, API key.
- Response headers: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`.
- Handling exceeded limits: return HTTP 429 (Too Many Requests) with `Retry-After` header.

### Example (Token Bucket with Redis)
```python
import time
import redis

class TokenBucket:
    def __init__(self, redis_client, key, capacity, refill_rate):
        self.redis = redis_client
        self.key = key
        self.capacity = capacity
        self.refill_rate = refill_rate  # tokens per second

    def allow_request(self):
        # Lua script for atomic update
        script = """
        local bucket = redis.call('hmget', KEYS[1], 'tokens', 'last_refill')
        local tokens = tonumber(bucket[1]) or 0
        local last_refill = tonumber(bucket[2]) or 0
        local now = tonumber(ARGV[1])
        local refill_rate = tonumber(ARGV[2])
        local capacity = tonumber(ARGV[3])

        local elapsed = now - last_refill
        local new_tokens = math.min(capacity, tokens + elapsed * refill_rate)

        if new_tokens >= 1 then
            redis.call('hmset', KEYS[1], 'tokens', new_tokens - 1, 'last_refill', now)
            return 1
        else
            return 0
        end
        """
        result = self.redis.eval(script, 1, self.key, time.time(), self.refill_rate, self.capacity)
        return bool(result)
```

### Real‑World Examples
- API gateways (Kong, AWS API Gateway) have built‑in rate limiting.
- Cloudflare rate limiting for DDoS protection.
- Stripe API rate limits by secret key.

## 🔁 How These Patterns Work Together

These patterns are not mutually exclusive; they complement each other:

- **Health checks** detect when a service is down.
- **Circuit breaker** prevents calls to a failing service and allows it to recover.
- **Retry with backoff** handles transient failures before they trip the circuit breaker.
- **Bulkhead** isolates failures so that a problem in one dependency doesn't exhaust resources for others.
- **Rate limiting** protects your service from being overwhelmed, complementing circuit breakers on the client side.

**Example flow:**
1. A service receives a request that requires calling an external API.
2. **Rate limiter** checks if the user has quota; if not, returns 429.
3. The call is attempted. If it fails with a 5xx error:
   - **Retry with backoff** kicks in, retrying a few times with increasing delays.
   - If failures persist, the **circuit breaker** trips to **Open**, failing fast for subsequent requests.
4. Meanwhile, **health checks** of the external API may indicate it's down, so the circuit breaker remains open until it's healthy again.
5. **Bulkheads** ensure that other parts of the system (e.g., calls to a different API) are not starved of threads.

## 📚 Summary

| Pattern | Purpose | Implementation Highlights |
|---------|---------|---------------------------|
| **Circuit Breaker** | Prevent repeated calls to a failing service | States: Closed, Open, Half‑Open; failure threshold; timeout |
| **Retry with Backoff** | Handle transient failures | Exponential backoff, jitter, max retries, retryable errors |
| **Bulkhead** | Isolate failures to prevent cascading | Thread pools, semaphores, connection pools |
| **Health Checks** | Determine if a service is alive/ready | Liveness vs readiness, HTTP endpoints, probes |
| **Rate Limiting** | Control request rate to protect resources | Algorithms: token bucket, leaky bucket, sliding window |

Mastering these reliability patterns enables you to design systems that are resilient, fault‑tolerant, and capable of gracefully handling failures – essential qualities for any large‑scale distributed application.
