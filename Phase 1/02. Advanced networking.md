## 🌐 WebSockets: Enabling Real-Time, Bidirectional Communication

WebSocket is a protocol that provides full-duplex communication channels over a single, long-lived TCP connection, diverging from the traditional HTTP request-response model .

### From HTTP Handshake to Persistent Connection
A WebSocket connection begins its life as a standard HTTP request. This is often called the **"WebSocket handshake"** . The client sends a request with special headers to "upgrade" the connection.
*   **Request Headers:** The client includes `Upgrade: websocket` and `Connection: Upgrade` to signal its intention .
*   **Server Response:** If the server supports WebSockets, it responds with a `101 Switching Protocols` status code, and the connection is "upgraded." From this point forward, the protocol switches from HTTP to WebSocket, and the persistent, bidirectional connection is established .

### Why WebSockets Matter in HLD
*   **Low Latency:** Once the connection is open, data (frames) can be sent from client to server or server to client instantly without the overhead of re-establishing a connection for each message. This is essential for real-time features .
*   **Efficiency:** It avoids the HTTP header overhead for each message, making it much more efficient for scenarios with a high frequency of small messages.

### Common HLD Use Cases
*   **Live Chat Applications (WhatsApp, Slack):** For instantly sending and receiving messages .
*   **Real-Time Dashboards & Live Scores:** Pushing updates to the UI as soon as data changes on the backend .
*   **Collaborative Editing (Google Docs):** Broadcasting changes from one user to all others currently viewing the document.
*   **Multiplayer Game Backends:** Synchronizing player actions and game state in real-time.

### Configuration Example: Nginx Reverse Proxy for WebSockets
When you place a reverse proxy like Nginx in front of your backend services, you must configure it to handle the connection upgrade correctly. Key directives include setting the correct `Upgrade` headers and increasing `proxy_read_timeout` to prevent the proxy from closing long-lived, idle WebSocket connections .
```nginx
map $http_upgrade $connection_upgrade {
    default upgrade;
    ''      close;
}

upstream ws_backend {
    server 127.0.0.1:8080;
    keepalive 64; # Enable connection keep-alive for performance
}

server {
    listen 443 ssl http2;
    server_name ws.example.com;

    keepalive_timeout 65s;

    location /ws/ {
        proxy_http_version 1.1; # Required for WebSocket upgrade
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection $connection_upgrade;

        # Crucial for long-lived connections: prevent proxy timeouts
        proxy_read_timeout 3600s;
        proxy_send_timeout 3600s;
        proxy_buffering off; # Disable buffering for real-time data

        proxy_pass http://ws_backend;
    }
}
```
*   The `map` block cleverly sets the `Connection` header to `upgrade` if an upgrade was requested, or `close` otherwise .
*   `proxy_http_version 1.1` is mandatory because the upgrade mechanism requires HTTP/1.1 .
*   `proxy_read_timeout` is set to a high value (e.g., 3600 seconds or 1 hour) to match the expected lifespan of a WebSocket connection .

### Caddy Example
Caddy handles WebSocket proxying transparently. Often, no special directives are needed; a standard `reverse_proxy` directive is sufficient .
```
example.com {
    reverse_proxy 127.0.0.1:8000 {
        transport http {
            read_timeout 2h # Increase timeout for long-lived WS connections
            write_timeout 2h
        }
    }
}
```

## ⚙️ gRPC: High-Performance RPC with HTTP/2

gRPC is a modern, open-source remote procedure call (RPC) framework developed by Google. It uses **HTTP/2** as its transport protocol and **Protocol Buffers** (protobuf) as its interface definition language and message serialization format .

### Why gRPC is a Game-Changer for Microservices
*   **Built on HTTP/2:** This is its superpower. It inherits all of HTTP/2's features, which are critical for performance .
*   **Protocol Buffers:** This binary serialization format is much faster and more compact than text-based formats like JSON, leading to lower latency and reduced network load.
*   **Strongly Typed APIs:** The `.proto` file clearly defines the service contract, which can be used to generate client and server code in numerous languages, ensuring type safety across services.

### Key gRPC Communication Patterns
1.  **Unary RPC:** Simple request-response, like a traditional HTTP call.
2.  **Server Streaming RPC:** The client sends one request and gets back a stream of responses (e.g., a news feed).
3.  **Client Streaming RPC:** The client sends a stream of requests and gets back a single response (e.g., uploading a large file in chunks).
4.  **Bidirectional Streaming RPC:** Both client and server send a stream of messages independently (e.g., a real-time chat or an interactive game).

### gRPC in High-Level Design
gRPC has become the go-to choice for communication between internal microservices due to its performance and efficiency .

### Configuration Example: Nginx Reverse Proxy for gRPC
Proxying gRPC traffic requires special handling. Nginx provides the `grpc_pass` directive specifically for this purpose.
```nginx
upstream grpc_backend {
    server 127.0.0.1:50051;
    keepalive 64; # gRPC benefits greatly from keep-alive
}

server {
    listen 443 ssl http2; # gRPC requires HTTP/2
    server_name api.example.com;

    # gRPC endpoint
    location / {
        grpc_pass grpc://grpc_backend; # Use grpc_pass, not proxy_pass
        grpc_read_timeout 3600s;
        grpc_send_timeout 3600s;
        grpc_set_header Host $host;
    }
}
```
*   The server block **must** include `http2` to accept HTTP/2 connections .
*   `grpc_pass` is used instead of `proxy_pass` to correctly route gRPC traffic .

### Caddy Example
Caddy also supports proxying gRPC, typically by specifying the `h2c` (HTTP/2 Cleartext) transport for the backend if it doesn't use TLS .
```
example.com {
    reverse_proxy 127.0.0.1:50051 {
        transport http {
            versions h2c # Specify HTTP/2 cleartext for the backend
        }
    }
}
```

## 🚀 HTTP/2: The Modern Web Protocol

HTTP/2 is a major revision of the HTTP protocol, designed to address the performance limitations of its predecessor, HTTP/1.1. It maintains the same semantics (methods, headers, status codes) but changes how data is formatted and transported between client and server .

### Key Features and How They Improve Performance
*   **Binary Protocol:** Unlike HTTP/1.1's text-based nature, HTTP/2 is binary. This makes it more efficient to parse, more compact over the network, and less error-prone .
*   **Multiplexing:** This is the most significant feature. In HTTP/1.1, only one request could be active at a time per connection (head-of-line blocking). HTTP/2 allows multiple requests and responses to be sent in parallel over a single TCP connection. This eliminates the need for domain sharding and dramatically improves page load times .
*   **Header Compression (HPACK):** HTTP/2 compresses request and response headers, which are often repetitive. This reduces overhead and latency, especially for requests with many headers.
*   **Server Push:** A server can proactively send resources to the client before the client requests them. For example, when a client requests an HTML page, the server could immediately push the associated CSS and JavaScript files.

### HTTP/2 in High-Level Design
*   **Enabler for gRPC:** As mentioned, gRPC relies on HTTP/2 for its advanced features like streaming and multiplexing .
*   **Load Balancers and Reverse Proxies:** Modern load balancers (like Nginx, Caddy, and Envoy) are typically configured to terminate HTTP/2 connections from clients and communicate with backends using either HTTP/1.1 or HTTP/2, offering a performance boost at the edge .

## 🔄 Reverse Proxies: The Gatekeepers of Your Backend

A reverse proxy is a server that sits between client devices and backend servers, intercepting requests and forwarding them to the appropriate server on behalf of the client. It acts as a single point of entry, abstracting the internal architecture of your application .

### Core Responsibilities in a System Design
*   **Load Balancing:** Distributing incoming traffic across a pool of backend servers to ensure no single server is overwhelmed .
*   **SSL/TLS Termination:** Decrypting incoming HTTPS requests and forwarding them as plain HTTP to internal services. This offloads the computational cost of encryption from the application servers .
*   **Caching:** Serving cached responses for static content or even dynamic content, reducing load on backend servers.
*   **Security & Anonymity:** Hiding the internal structure of your network and providing a single point to enforce security policies, rate limiting, and block malicious requests .
*   **Request Routing:** Directing requests to different services based on the request's path, headers, or other attributes (a core function of API gateways) .

### Architecture Example: Proxying Multiple Services
This example from  shows how a reverse proxy can route different types of traffic to the same backend application:
*   `https://dashboard.example.com/` -> Serves the main Web UI.
*   `https://dashboard.example.com/api/v1/ws/` -> Upgraded to a WebSocket connection for real-time terminal access.
*   `https://dashboard.example.com/proto.NezhaService/` -> Proxied as gRPC traffic for agent communication.

### Configuration Example: Caddy with Load Balancing and Health Checks
Caddy's `reverse_proxy` directive offers a clean way to configure advanced features like load balancing policies and active health checks .
```
example.com {
    reverse_proxy 10.0.0.11:8000 10.0.0.12:8000 10.0.0.13:8000 {
        lb_policy round_robin  # Distribute requests in a rotating fashion
        health_uri /health      # Health check endpoint
        health_interval 10s
        health_timeout 2s
        fail_duration 30s       # Mark backend as down for 30s after a failed check

        transport http {
            dial_timeout 5s
            read_timeout 30s
            write_timeout 30s
        }
    }
}
```

## 🌍 CDNs (Content Delivery Networks): Distributing Content Globally

A Content Delivery Network (CDN) is a geographically distributed group of servers that work together to provide fast delivery of Internet content. A CDN allows for the fast transfer of assets needed for loading Internet content, including HTML pages, JavaScript files, stylesheets, images, and videos .

### How a CDN Works
1.  **Caching at the Edge:** Content is cached on proxy servers (Points of Presence or PoPs) located in data centers around the world.
2.  **Geographic Routing:** When a user requests your content, the CDN routes the request to the server geographically closest to them (the "edge"). This minimizes latency .
3.  **Origin Offload:** By serving cached content from the edge, the CDN dramatically reduces the load on your origin servers.

### The Role of CDNs in Modern HLD
*   **Improved Load Times:** By serving content from a location near the user.
*   **Reduced Bandwidth Costs:** Caching and compression reduce the amount of data served from the origin.
*   **Increased Content Availability and Redundancy:** Distributed architecture can handle high traffic and hardware failures better than a single origin server.
*   **Enhanced Security:** CDNs can provide DDoS mitigation, Web Application Firewalls (WAF), and other security services.

### CDNs and Advanced Protocols
When using a CDN with advanced protocols, you need to be aware of their capabilities .
*   **WebSockets:** Many CDNs, like Cloudflare on a free plan, support proxying WebSocket connections. However, heavy usage might violate terms of service. Gcore also supports WebSockets.
*   **gRPC:** Support is less universal. Cloudflare supports gRPC, but Gcore, for example, does not on its free tier .
*   **Configuration:** You typically need to ensure your CDN is set to "proxied" mode (the orange cloud in Cloudflare) and that SSL/TLS settings are correctly configured (e.g., `Full` or `Full (strict)`) to maintain end-to-end encryption .

### Real-World Example: V2Ray with CDN
A common pattern for evading network restrictions is to place a CDN in front of a proxy server like V2Ray. The client connects to your domain via the CDN, which then forwards the WebSocket or gRPC traffic to your origin server. This masks the origin server's IP address and makes the traffic appear as normal web traffic .

This integrated set of technologies forms the backbone of many high-performance, scalable, and resilient systems you'll design.
