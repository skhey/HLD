# NoSQL Databases: Types, Use Cases, and Trade-Offs

NoSQL databases emerged to address limitations of relational databases in handling massive scale, flexible schemas, and specific data access patterns. They are categorized into four main types: **Key-Value**, **Document**, **Wide-Column**, and **Graph**. Each type optimizes for different workloads and comes with its own set of trade-offs.

---

## 1️⃣ Why NoSQL?

- **Scalability:** Designed to scale horizontally (distribute data across many servers) by nature.
- **Flexible Schema:** Allow semi-structured or unstructured data, enabling faster iteration.
- **Performance:** Optimized for specific access patterns (e.g., key-based lookups, graph traversals).
- **High Availability:** Often built with replication and partition tolerance as first-class citizens (AP systems in CAP).

**CAP Trade-Off:** Most NoSQL databases prioritize **Availability** and **Partition Tolerance** over strong consistency, offering eventual consistency. Some allow tunable consistency (e.g., Cassandra).

---

## 2️⃣ Key-Value Stores

### Characteristics
- Simplest NoSQL model: data stored as a collection of key-value pairs.
- Keys are unique, values can be strings, JSON blobs, or binary data.
- Operations: `GET(key)`, `PUT(key, value)`, `DELETE(key)`.
- No query language; access is solely by key.

### Examples
- **Redis:** In-memory with optional persistence; supports data structures (lists, sets, hashes).
- **Amazon DynamoDB:** Managed, highly available; supports both key-value and document models.
- **Memcached:** Pure in-memory cache.
- **Riak:** Distributed, fault-tolerant.

### Use Cases
- **Caching:** Store frequently accessed data (e.g., user sessions, HTML fragments).
- **Session Store:** Maintain user session state in web applications.
- **Shopping Cart:** Simple key-value pairs (cart ID → cart contents).
- **User Profiles:** When access is by user ID.
- **Leaderboards:** Using Redis sorted sets.

### Trade-Offs
| Pros | Cons |
|------|------|
| Extremely fast reads/writes (often in-memory) | Limited querying – cannot query by value or range without additional indexes |
| Simple data model, easy to scale | No relationships or joins; must handle at application level |
| High throughput and low latency | Values are opaque; cannot query inside complex structures |
| Great for caching and session management | Data size may be limited by memory (for in-memory stores) |

### Example: Redis for Session Store
```redis
# Store user session with TTL 1 hour
SET session:12345 '{"user_id":123, "name":"Alice"}' EX 3600
# Retrieve
GET session:12345
```

## 3️⃣ Document Stores

### Characteristics
- Stores data as **documents** (JSON, BSON, XML, etc.).
- Documents have a structure (fields) but schema is flexible; fields can vary across documents.
- Documents are grouped into **collections** (analogous to tables).
- Supports secondary indexes, rich queries (including field-level queries, ranges, aggregations).

### Examples
- **MongoDB:** Most popular; uses BSON; supports replication, sharding, secondary indexes.
- **CouchDB:** Uses JSON; focuses on availability and conflict resolution (multi-master).
- **Amazon DocumentDB:** MongoDB-compatible managed service.
- **Firebase Firestore:** Real-time sync, mobile-friendly.

### Use Cases
- **Content Management:** Blog posts, articles, product catalogs with varying attributes.
- **User Profiles:** Each user can have different fields.
- **Real-Time Feeds:** Store and query activity streams.
- **E-commerce Catalogs:** Products with diverse specifications.
- **Logging / Event Storage:** Each event as a document.

### Trade-Offs
| Pros | Cons |
|------|------|
| Flexible schema – adapts to changing requirements | Eventual consistency in many systems (though tunable) |
| Rich query language (MongoDB has powerful aggregations) | Document size limits (e.g., MongoDB 16MB per document) |
| Horizontal scaling via sharding | Joins are not native; need embedded documents or application-side joins |
| Good performance for nested data | May require denormalization to avoid multiple queries |

### Example: MongoDB Document
```json
{
  "_id": ObjectId("507f1f77bcf86cd799439011"),
  "username": "alice",
  "email": "alice@example.com",
  "profile": {
    "age": 30,
    "country": "USA"
  },
  "orders": [
    { "order_id": "123", "total": 99.99 },
    { "order_id": "456", "total": 149.99 }
  ]
}
```

## 4️⃣ Wide-Column Stores

### Characteristics
- Inspired by Google Bigtable.
- Data stored in tables with rows and columns, but columns can be dynamic and grouped into **column families**.
- Each row can have a different set of columns.
- Designed for massive scale, high write throughput, and time-series data.
- Often use **sorted order** by row key for efficient range scans.

### Examples
- **Apache Cassandra:** Peer-to-peer, high availability, tunable consistency.
- **HBase:** Runs on Hadoop/HDFS, strong consistency, good for batch processing.
- **ScyllaDB:** Cassandra-compatible, written in C++ for lower latency.
- **Google Bigtable:** Managed wide-column database.

### Use Cases
- **Time-Series Data:** IoT sensor readings, server metrics (each row key includes timestamp).
- **User Activity Logs:** Write-heavy, need to store large volumes.
- **Recommendation Engines:** Store user-item interactions.
- **Messaging Systems:** Like Cassandra for chat applications.
- **Large-Scale Analytics:** With Hadoop/HBase.

### Trade-Offs
| Pros | Cons |
|------|------|
| Extremely high write throughput | Complex data modeling (requires understanding of partition keys, clustering columns) |
| Linear scalability – just add nodes | No multi-row transactions (ACID) unless using lightweight transactions (Cassandra) |
| Tunable consistency (e.g., Cassandra: ANY, ONE, QUORUM, ALL) | Querying limited to primary key patterns; secondary indexes have performance caveats |
| Built for fault tolerance and replication | Learning curve – CQL (Cassandra Query Language) similar to SQL but differences |

### Example: Cassandra Table for Time-Series
```cql
CREATE TABLE sensor_data (
  sensor_id UUID,
  timestamp TIMESTAMP,
  value DOUBLE,
  PRIMARY KEY (sensor_id, timestamp)
) WITH CLUSTERING ORDER BY (timestamp DESC);
-- This stores data for each sensor, sorted by timestamp descending.
-- Query: get latest 10 readings for a sensor
SELECT * FROM sensor_data WHERE sensor_id = ? LIMIT 10;
```

## 5️⃣ Graph Databases

### Characteristics
- Data stored as **nodes** (entities) and **edges** (relationships) with properties.
- Optimized for traversing relationships efficiently.
- Use graph query languages (Cypher, Gremlin, SPARQL).

### Examples
- **Neo4j:** Most popular; ACID compliant; uses Cypher.
- **Amazon Neptune:** Managed, supports both RDF and property graph models.
- **ArangoDB:** Multi-model (graph + document).
- **JanusGraph:** Distributed, integrates with Cassandra/HBase.

### Use Cases
- **Social Networks:** Friends, followers, recommendations.
- **Fraud Detection:** Identify suspicious patterns (e.g., circular transactions).
- **Recommendation Engines:** "People who bought X also bought Y."
- **Network & IT Operations:** Dependency graphs, infrastructure mapping.
- **Knowledge Graphs:** Semantic search, data lineage.

### Trade-Offs
| Pros | Cons |
|------|------|
| Intuitive modeling for connected data | Not suitable for massive scans of all nodes; graph queries are deep but not wide |
| Extremely fast relationship traversals (index-free adjacency) | Can become complex to shard (graph partitioning is hard) |
| Support for complex graph algorithms (shortest path, community detection) | Smaller ecosystem compared to SQL or document stores |
| ACID possible in some (Neo4j) | May not be ideal for high-volume simple key-value lookups |

### Example: Neo4j Cypher Query
```cypher
// Find friends of friends of Alice
MATCH (alice:Person {name: 'Alice'})-[:FRIEND]->()-[:FRIEND]->(fof)
RETURN fof.name
```

## 6️⃣ Comparison Table

| Feature | Key-Value | Document | Wide-Column | Graph |
|---------|-----------|----------|-------------|-------|
| **Data Model** | Key → Value | JSON/BSON documents | Tables with column families | Nodes & Edges |
| **Query Capabilities** | By key only | Rich (field-level, aggregations) | By primary key, secondary indexes (limited) | Relationship traversal, graph algorithms |
| **Schema** | None | Flexible | Flexible (column families) | Flexible (properties) |
| **Scaling** | Horizontal (sharding) | Horizontal (sharding) | Horizontal (partitioning by key) | Vertical scaling common; sharding complex |
| **Consistency** | Tunable (e.g., DynamoDB eventually consistent) | Tunable (MongoDB allows strong reads) | Tunable (Cassandra: eventual to strong) | ACID in some (Neo4j) |
| **Best For** | Caching, sessions | Catalogs, content, real-time apps | Time-series, event logging, high write throughput | Connected data, relationships |
| **Examples** | Redis, DynamoDB | MongoDB, CouchDB | Cassandra, HBase | Neo4j, Neptune |

## 7️⃣ Choosing Between SQL and NoSQL (and Which NoSQL)

| Use Case | Recommended Choice | Reasoning |
|----------|-------------------|-----------|
| Financial transactions, inventory with strict ACID | **SQL** | Strong consistency, complex transactions, joins. |
| User profiles with varying attributes | **Document** | Flexible schema, easy to evolve. |
| Real-time leaderboard, session cache | **Key-Value** | Ultra-low latency, simple access. |
| IoT sensor data, event logging | **Wide-Column** | High write throughput, time-series optimized. |
| Social network, recommendation engine | **Graph** | Relationship-heavy queries. |
| Product catalog with many attributes and categories | **Document** (or SQL with JSON) | Flexible schema, can embed related data. |
| Large-scale analytics (batch) | **Wide-Column** (HBase) or SQL on Hadoop | Designed for big data. |

### NoSQL Design Considerations
- **Denormalization:** Often necessary to avoid joins.
- **Data Duplication:** May store same data in multiple places for query efficiency.
- **Consistency Trade-Offs:** Choose tunable consistency levels based on business needs.
- **Query Patterns:** Design your data model around how you will query (not around relationships as in SQL).

---

## 8️⃣ Key Takeaways for HLD Interviews

- **NoSQL is not a replacement for SQL** – they coexist and serve different purposes.
- **Understand the four types** and their strengths/weaknesses.
- **Use cases:** Be able to justify why a particular NoSQL database fits a scenario.
- **Trade-offs:** Always discuss consistency, queryability, and scalability implications.
- **Real-world examples:** Mention specific databases (e.g., Cassandra for write-heavy, Redis for caching, Neo4j for graphs).

In system design, you often combine multiple databases (polyglot persistence) – e.g., SQL for orders, Redis for sessions, Cassandra for logs, Neo4j for fraud detection.

Mastering NoSQL patterns will allow you to design systems that scale elegantly and meet diverse requirements.
