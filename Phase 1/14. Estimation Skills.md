# Estimation Skills: DAU, QPS, Storage, Bandwidth Calculations

In system design interviews, estimation (also called "back‑of‑the‑envelope calculations") is a critical skill. It demonstrates your ability to think about scale, make reasonable assumptions, and translate user activity into concrete infrastructure requirements. This guide covers the key metrics, formulas, and step‑by‑step examples you need to master.

---

## 1️⃣ Why Estimation Matters

- **Informs architectural decisions:** Knowing expected QPS helps choose between SQL/NoSQL, decide on caching, and plan for sharding.
- **Validates feasibility:** Estimates reveal whether a design can handle the load (e.g., “a single server can handle 10k QPS? Probably not.”).
- **Shows structured thinking:** Interviewers want to see how you break down a problem and make reasonable assumptions.

**Remember:** The goal is not extreme precision, but order‑of‑magnitude accuracy. Being off by a factor of 2 is fine; being off by a factor of 1000 is not.

---

## 2️⃣ Key Metrics and Units

| Metric | Description | Common Units |
|--------|-------------|--------------|
| **DAU** | Daily Active Users | millions |
| **MAU** | Monthly Active Users | millions |
| **QPS** | Queries Per Second (or Requests Per Second) | requests/sec |
| **Peak QPS** | Maximum requests per second during busy hours | often 2–3× average |
| **Storage** | Total data to be stored | GB, TB, PB |
| **Bandwidth** | Data transferred per second | MB/s, Gbps |
| **Cache Size** | Amount of hot data to cache | GB, TB |

### Common Conversions
- 1 million seconds ≈ 11.6 days (use for daily to per‑second conversions)
- 1 PB = 1024 TB = ~10⁶ GB
- 1 Gbps = 125 MB/s

## 3️⃣ Traffic Estimation (DAU → QPS)

The first step is to estimate the number of requests per second your system must handle.

### Formula
```
Average QPS = (DAU × Actions per user per day) / (24 × 3600)
Peak QPS = Average QPS × Peak Factor
```

**Where:**
- **DAU:** Estimated from total users and engagement (e.g., 10% of MAU are active daily).
- **Actions per user per day:** For a social media app, this could be number of posts read, written, etc. (e.g., 10 reads, 1 write).
- **Peak Factor:** Typically 2–3 for daily patterns; sometimes 5–10 for special events (e.g., Super Bowl).

### Example: Twitter Timeline Requests
- Assume 300M DAU.
- Each user views their timeline 5 times per day (reads).
- Each user posts 1 tweet per day (writes).

**Reads per day:** 300M × 5 = 1.5B reads  
**Writes per day:** 300M × 1 = 300M writes  
**Total requests:** 1.8B per day

**Average QPS:** 1.8B / (24 × 3600) ≈ 1.8B / 86,400 ≈ 20,800 QPS  
**Peak QPS:** Assume 3× average = ~62,400 QPS

*Note:* This is only for timeline and tweet endpoints. Other endpoints (search, login) add more.

## 4️⃣ Storage Estimation

Estimate how much data will accumulate over a retention period (e.g., 5 years).

### Formula
```
Total Storage = (Number of items) × (Size per item) × (Replication Factor)
Number of items = (Daily new items) × (Retention days)
```

**Where:**
- **Daily new items:** e.g., new tweets, photos, messages.
- **Size per item:** Average size of a tweet (including metadata), a photo, etc.
- **Replication factor:** For fault tolerance, data is often replicated (e.g., 3× in Cassandra, 2× in cloud storage).
- **Backup factor:** Optional additional copies for disaster recovery.

### Example: Twitter Tweet Storage
- **Tweets per day:** 500M (from earlier 300M users × 1.67 tweets/user).
- **Average tweet size:** 280 characters ≈ 280 bytes. But with metadata (user ID, timestamp, media links, etc.), assume 1 KB.
- **Retention:** 5 years = 1825 days.
- **Replication:** For high availability, assume 3 replicas in the database (e.g., Cassandra).

**Daily new storage:** 500M × 1 KB = 500 GB/day  
**5‑year raw storage:** 500 GB × 1825 ≈ 912 TB  
**With replication (3×):** 912 TB × 3 ≈ 2.74 PB

**Note:** This is only tweet content. Add indexes, logs, etc., which can double or triple the total.

### Example: YouTube Video Storage
- **Videos uploaded per day:** 500,000 hours (public stat: ~500 hours per minute → 720k hours/day). We'll use 500k hours.
- **Average video bitrate:** Assume 5 Mbps (720p).
- **Size per hour:** (5 Mbps × 3600 seconds) / 8 = 2.25 GB/hour.
- **Daily raw video:** 500k × 2.25 GB = 1.125 PB/day.
- **Retention:** Forever, but compress older videos, store multiple qualities. For simplicity, 5 years: 1.125 PB × 1825 ≈ 2,053 PB = 2 EB (exabytes).
- **Replication:** Typically 3 copies → 6 EB.

*This illustrates why YouTube needs massive storage and CDN.*

## 5️⃣ Bandwidth Estimation

Bandwidth is the rate of data transfer. It matters for network capacity and cost.

### Formula
```
Bandwidth (MB/s) = (Daily data transferred) / (24 × 3600)
or
Bandwidth = (QPS) × (Average response size)
```

**Where:**
- **Daily data transferred:** Sum of all data sent to clients (egress) and received from clients (ingress).
- **Average response size:** For a timeline request, maybe 1 KB of JSON plus images? But images are usually separate CDN.

### Example: Twitter Bandwidth (Text‑Only Timeline)
- Assume timeline requests return 20 tweets, each 1 KB → 20 KB per response.
- Reads per day: 1.5B → 1.5B × 20 KB = 30 TB/day.
- **Egress bandwidth (text):** 30 TB / 86400 ≈ 350 MB/s.
- Plus media (images, videos) served from CDN, which would be much higher.

### Example: WhatsApp Message Bandwidth
- Messages per day: 100B (public stat: ~100B messages/day).
- Average message size (text): 100 bytes.
- Media messages (images, video) are much larger but fewer. Let's separate.
- **Text egress:** 100B × 100 B = 10 TB/day ≈ 115 MB/s.
- Media: Assume 1B images/day, average 200 KB each → 200 TB/day ≈ 2.3 GB/s.

*This shows why messaging apps heavily compress media and use CDNs.*

## 6️⃣ Caching Estimation

Estimate how much memory you need for caching to reduce database load.

### Formula
```
Cache Size = (Number of active items) × (Item size) × (Overhead)
```

**What to cache?** Typically the working set – data accessed frequently. For social media, you might cache the last N days of posts per user.

### Example: Twitter Timeline Cache
- Each user's timeline is pre‑computed. If you store last 100 tweets for 300M users, each timeline entry is a tweet ID (8 bytes) + timestamp (8 bytes) ≈ 16 bytes per tweet.
- **Size per timeline:** 100 × 16 = 1.6 KB.
- **Total cache if store all users' timelines:** 300M × 1.6 KB = 480 GB. This is feasible in Redis (several nodes).
- Plus overhead for Redis structures, maybe 2× → ~1 TB.

### Cache Hit Ratio
- You can estimate the percentage of requests that hit cache. If 99% of timeline reads hit cache, database load is reduced to 1% of reads.

## 7️⃣ Practice Examples with Step‑by‑Step Solutions

### Example 1: URL Shortener (bit.ly)
**Requirements:**
- 100M new short URLs per month.
- 10:1 read‑to‑write ratio (i.e., each short URL is read 10 times on average).
- Retention: indefinite.
- Average URL length: 100 characters, short ID: 7 characters.

**Estimates:**

- **Daily new URLs:** 100M / 30 ≈ 3.33M per day.
- **Writes per second:** 3.33M / 86400 ≈ 38.5 writes/sec.
- **Reads per second:** 38.5 × 10 = 385 reads/sec.
- **Peak QPS:** 2× average = 770 reads/sec, 77 writes/sec.

- **Storage (5 years):**
  - Each mapping: short ID (7 chars) + original URL (100 chars) + metadata (timestamp, user ID) → assume 200 bytes.
  - Daily new storage: 3.33M × 200 B = 666 MB/day.
  - 5 years: 666 MB × 1825 ≈ 1.2 TB.
  - Replication (3×) = 3.6 TB.
  - Indexes add overhead (maybe 2×) = 7.2 TB. Manageable.

- **Bandwidth (reads):** 385 reads/sec × 200 B = 77 KB/s (trivial). But if you consider redirects (HTTP 301/302), response size is tiny. So bandwidth is negligible.

### Example 2: WhatsApp Messaging
**Assumptions:**
- 2B MAU, 1.5B DAU.
- Each user sends 40 messages/day (text) and 2 media messages/day.
- Text message size: 100 bytes.
- Media: image 200 KB (after compression), video 2 MB (average).
- Retention: 30 days for messages (ephemeral) – but media may be stored longer.

**Traffic:**
- Daily text messages: 1.5B × 40 = 60B messages.
- Daily media messages: 1.5B × 2 = 3B (mostly images, some videos).
- **Text writes/sec:** 60B / 86400 ≈ 694k writes/sec.
- **Media writes/sec:** 3B / 86400 ≈ 34.7k writes/sec (mostly images).
- Reads: Assume each user reads 50 messages/day (includes viewing media). Reads = 1.5B × 50 = 75B reads/day → 868k reads/sec.

**Storage (30 days):**
- Text storage: 60B × 100 B = 6 TB/day × 30 = 180 TB.
- Media storage (images): 2.5B images/day × 200 KB = 500 TB/day × 30 = 15 PB.
- Media storage (videos): 0.5B videos/day × 2 MB = 1 PB/day × 30 = 30 PB.
- Total raw storage for 30 days: ≈ 45 PB.
- Replication (3×) = 135 PB. This is why WhatsApp deletes old messages and uses highly compressed storage.

**Bandwidth:**
- Egress for reads: 75B messages × 100 B = 7.5 TB/day → 87 MB/s (text).
- Media egress: Assume 3B media reads/day × average 200 KB = 600 TB/day → 6.9 GB/s. This would overwhelm a single data center, hence CDN and edge caching.

### Example 3: Netflix Streaming
**Assumptions:**
- 200M subscribers.
- Average viewing per user: 2 hours/day.
- Average bitrate: 5 Mbps (720p/1080p).
- Peak factor: 3× (evenings).

**Bandwidth:**
- Daily streaming seconds: 200M × 2 × 3600 = 1.44 × 10¹² seconds.
- Daily data transferred: 1.44e12 sec × 5 Mbps / 8 = 9 × 10¹¹ MB = 900 PB/day.
- Average egress bandwidth: 900 PB / 86400 ≈ 10.4 TB/s = 83 Tbps.
- Peak bandwidth: 3× = 250 Tbps.

This is why Netflix relies heavily on CDN (Open Connect) to cache content close to users.

## 8️⃣ Common Pitfalls

- **Forgetting replication and overhead:** Storage estimates must include replication (usually 2–3×) and metadata overhead.
- **Confusing bytes and bits:** Bandwidth is often in bits per second (bps), storage in bytes. Convert carefully.
- **Ignoring peak factor:** Designing for average leads to under‑provisioning.
- **Assuming all data is equal:** Large objects (videos, images) dominate storage and bandwidth.
- **Not accounting for growth over time:** Estimate for a reasonable horizon (3–5 years).

---

## 9️⃣ Tips for Interviews

- **State your assumptions clearly.** The interviewer cares more about your reasoning than the exact numbers.
- **Round numbers for simplicity.** Use 1M = 10⁶, 1B = 10⁹, etc.
- **Use order‑of‑magnitude checks:** If your estimated QPS is 10¹², that's obviously wrong (faster than light).
- **Practice with different scenarios:** Social media, video, file storage, etc.

## 📚 Summary

| Metric | Formula | Example |
|--------|---------|---------|
| **QPS** | (DAU × actions) / 86400 | 300M × 5 / 86400 ≈ 17k QPS |
| **Storage** | (daily items × size × retention) × replication | 500M × 1KB × 1825 × 3 ≈ 2.7 PB |
| **Bandwidth** | (QPS × response size) or (daily data / 86400) | 17k × 20KB ≈ 340 MB/s |

Mastering estimation builds confidence in your designs and shows interviewers you can think at scale. Practice with real‑world numbers from known services (Twitter, WhatsApp, YouTube) and adjust assumptions to fit different scenarios.
